#  网络同步系统介绍

绿洲启元的网络同步主要基于UE4引擎的同步系统，官方文档可以了解到该同步系统的基本知识。<br>
[https://docs.unrealengine.com/zh-CN/InteractiveExperiences/Networking/Overview/index.html](https://docs.unrealengine.com/zh-CN/InteractiveExperiences/Networking/Overview/index.html)

# 属性同步：

## 介绍：

可以对Actor中的参数进行同步，同步规则为从服务器单向同步至客户端，客户端的参数无法同步至服务器，同步方式为每隔1-X帧自动进行检测参数是否有改变（可以在Actor的NetPrioity参数中设置检测频率），把改变的参数同步至客户端。
同步开销较小，合适由DS确定，并需要持续修改且并不依赖时效性的参数，如人物属性，比赛比分等参数。需要注意的是由于是一段时间进行一次检测，如果在一帧多次修改同步参数，中间的修改会丢失，取最后一次修改的值进行同步。

## 使用方式：

1.开启同步，Actor属性面板中搜索到Replicates参数，勾选，或者在Lua脚本中调用AActor.SetReplicates函数设置<br>
2.注册同步变量，Lua脚本中新增类函数 Actor:GetReplicatedProperties()，并返回需要同步如躲猫猫案例：

```lua
local PeekabooPlayerController = 
{
    IsTransform = false;
    TransformTypes = {};
}
function PeekabooPlayerController:GetReplicatedProperties()
    return 
    "IsTransform",
    "TransformTypes"
end
```

3.监听同步事件，同步事件接口名为OnRep_变量名()，参考躲猫猫案例：

```lua
function PeekabooPlayerController:OnRep_IsTransform()
    print(string.format("PeekabooPlayerController:OnRep_IsTransform[%s] IsTransform[%s]", self.PlayerName, tostring(self.IsTransform)));

    UGCEventSystem:SendEvent(PeekabooModeEventType.IsTransformChange, self, self.IsTransform);
end

function PeekabooPlayerController:OnRep_TransformTypes()
    print(string.format("PeekabooPlayerController:OnRep_TransformTypes[%s]", self.PlayerName));
    log_tree("TransformTypes:",self.TransformTypes);

    UGCEventSystem:SendEvent(PeekabooModeEventType.TransformTypesChange, self, self.TransformTypes);
end
```

## 注意事项：

属性同步的OnRep响应事件有先后顺序而且无法保证顺序，所以同步参数不能存在相互的强依赖，如有强依赖的参数，可以放在一个LuaTable中进行同步。

# RPC

## 介绍：

RPC是一种服务器向客户端，或者客户端向服务器发送事件的同步方法，可以向另一端发送事件并携带所需参数。

## CallUnrealRPC

从服务器端或者客户端发起，在另一端响应的RPC，也就是DS调用会发送到客户端，在客户端调用会发送至服务器，分为可靠和不可靠类型。<br>
可靠RPC有校验机制能保证不会因为网络原因丢包导致另一端没有收到，代价则是开销更大，合适低频率发送但是要求必须收到的事件，如各种Tips。<br>
不可靠RPC则无法保证，但是同时也不容易造成网络阻塞。合适高频发送但是允许丢失的事件，如客户端向服务器发送一些高频流水数据，或者各种临时表现型的RPC（播放动画，粒子），请根据需要选择合适的方案。

### 使用方式：

可靠RPC：UnrealNetwork.CallUnrealRPC(TargetPlayerController, TargetActor, FunctionName, ...)<br>
不可靠RPC：UnrealNetwork.CallUnrealRPC_Unreliable(TargetPlayerController, TargetActor, FunctionName, ...)<br>
1.确认RPC发送端，如想向DS发送ServerRPC，在Controller或者其他Owner为Controller的Actor中增加RPC事件接口，可参考躲猫猫案例：

```
--申请点赞玩家
function PeekabooPlayerController:ServerRPC_LikePlayerByPlayerKey(Playerkey)
    print(string.format("PeekabooPlayerController:ServerRPC_LikePlayerByPlayerKey[%s], Playerkey[%d]", self.PlayerName, Playerkey));
    ......
end
```

2.注册ServerRPC，如果为客户端向DS发送的ServerRPC，需要在GetAvailableServerRPCs事件中进行注册，DS向客户端发送的ClientRPC无需注册。

```
function PeekabooPlayerController:GetAvailableServerRPCs()
    return 
    "ServerRPC_ReRandomTransformTypes",
    "ServerRPC_SelectTransformType",
    "ServerRPC_LikePlayerByPlayerKey";
end
```

3.发送RPC，在客户端需要发送的地方，调用 UnrealNetwork.CallUnrealRPC/CallUnrealRPC_Unreliable。

```
--点赞按钮按下
function PeekabooPlayerlistItem:Button_Like_OnClicked()
    .......
    --发送点赞申请
    UnrealNetwork.CallUnrealRPC(PeekabooMode.OwnerController, PeekabooMode.OwnerController, "ServerRPC_LikePlayerByPlayerKey", self.PlayerResultData.PlayerKey);
     ......
end
```

## CallUnrealRPC_Multicast

从服务器端发起，广播到所有持有该Actor的客户端中的广播事件，和CallUnrealRPC类似，同样分为可靠和不可靠类型，适用于需要每一个客户端或者相关客户端都收到事件的情况下使用，比如击杀播报。

### 使用方式：

CallUnrealRPC_Multicast(TargetActor, FunctionName, ...)
CallUnrealRPC_Multicast_Unreliable(TargetActor, FunctionName, ...)
可以参考躲猫猫案例中的游戏结算事件：

```
function Action_GameSettlement:SendGameResultData()
    local GameResultData = {};
    ......
    UnrealNetwork.CallUnrealRPC_Multicast(g_current_gamestate ,"MulticastRPC_SendEvent", PeekabooModeEventType.GameEnd, GameResultData);
end

--向所有客户端广播事件
function PeekabooGameState:MulticastRPC_SendEvent(EventKey, ...)
    UGCEventSystem:SendEvent(EventKey, ...);
end
```

## 注意事项：

在同一个Actor中发送的CallUnrealRPC事件，在另一端接收可以保证接收顺序，但是在不同Actor中发送的RPC事件则无法保证接收顺序，如需严格保序的数据，建议在同一个Actor中进行发送。

#  使用枚举类型

绿洲启元编辑器支持枚举类型，但绿洲启元编辑器枚举类型的创建和使用与虚幻引擎稍有不同，下面会逐步介绍。
<br>
## 基本介绍

### 1. 枚举是什么？

![企业微信截图_16868353788232.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/%E4%BC%81%E4%B8%9A%E5%BE%AE%E4%BF%A1%E6%88%AA%E5%9B%BE_16868353788232.png)

**枚举** （英：Enum或Enumeration）用于为整数值命名，通常用于在复数状态下（≥3）进行分类判断（对应布尔值仅支持0和1）。

例如，一个基本的游戏菜单状态可以表示为名称并编程为整数。

```js
– 0 = Mainmenu
– 1 = Game
– 2 = Pause
– 3 = Gameover
```

### 2. 绿洲枚举类型与虚幻原生枚举类型对比

所有游戏引擎都可以使用枚举，因为它们是基本的编程功能。虚幻引擎可以在c++和蓝图中使用枚举，绿洲启元编辑器继承了此特性，并支持在lua中使用枚举。<br>

| 项名 | 编辑器中创建 | 不允许与系统枚举重名 | 不允许创建同名枚举 | lua中添加自动映射 |
| ------ | ------ | ------ | ------ | ------ |
| 虚幻 | ✔ | ✖ | ✖ | ✖ |
| 绿洲（2022年4月版本） | ✔ | ✖ | ✔ | ✖ |
| 绿洲（2022年6月版本） | ✔ | ✔ | ✔ | ✔ |

<br>

## 在绿洲启元编辑器下创建枚举

### 1. 创建枚举

1.1 在【内容浏览器】的空白处右击，找到`蓝图` -`枚举` ，创建一个枚举类型产并为其命名（如“MyEnum”）。

![企业微信截图_16868353906119.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/%E4%BC%81%E4%B8%9A%E5%BE%AE%E4%BF%A1%E6%88%AA%E5%9B%BE_16868353906119.png)
  
1.2  枚举命名注意

- 避免与系统保留枚举重名：[此API页面](https://developer.gp.qq.com/api/#/?classDetailShow=false&isSelect=1)记录了所有和平**全局保留的枚举类型**，请开发者在创建自定义枚举类型时避免与此页面下的枚举重名（2022年6月版本绿洲启元编辑器固定不允许玩家创建与系统同名的枚举）。
- 避免同一个工程下重名：同一个玩法工程下也不允许创建**同名枚举**，即便同名的枚举分别在不同的目录下。

双击创建好的枚举类型资产，打开枚举值编辑界面，点击【新建】按钮为创建的枚举添加不同的枚举值，并为其命名和添加描述，点击保存来应用更改

### 2. 使用枚举

可参照如下示例使用枚举类型

2.1 蓝图设置

先创建一个要调用枚举值的Actor（如“MyActor”）。若要以蓝图变量方式存储枚举值，可在创建的Actor下新建一个变量，类型为用户自行创建的枚举类型（如“MyEnum”），你可以随时更改此变量的枚举值。

`注意：以蓝图变量方式存储枚举值非必须选项，开发者可根据具体需求选择是否需要以蓝图变量方式存储枚`

![企业微信截图_16868354621148.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/%E4%BC%81%E4%B8%9A%E5%BE%AE%E4%BF%A1%E6%88%AA%E5%9B%BE_16868354621148.png)

2.2 lua脚本设置

你也可以在lua脚本中定义变量以存储和改变枚举值。

```lua
--Edit Below--
local MyActor = {
    MyEnumration = MyEnum.Banana
};
```

枚举值在计算和使用时**被视为整数**，例如定义如下枚举：

```lua
---@enum MyEnum
MyEnum = { 
    Apple = 0,
    Orange = 1,
    Banana = 2,
};
```

可通过如下代码打印：

```lua
function MyActor:ReceiveBeginPlay()
    print("MyActor:ReceiveBeginPlay");
    print(MyEnum.Apple); -- 输出的值为0
end
```

这里贴出完整的示例工程代码，方便开发者参考使用，另附示例工程压缩包，请前往文章末尾的附件进行下载。

```lua
---@class MyActor_C:Actor
---@field Box UBoxComponent
---@field DefaultSceneRoot USceneComponent
---@field MyEnumeration TEnumAsByte<MyEnum>

require('Script.Common.ue_enum_custom') -- 2022年6月版本无需手动require

--Edit Below--
local MyActor = {}; 

function MyActor:ReceiveBeginPlay()
    print("MyActor:ReceiveBeginPlay");
    print(MyEnum.Apple); -- 输出的值为0
    print(self.MyEnumeration); -- 输出的值与MyActor蓝图中的枚举值一致
    if self:HasAuthority() == true then
        --仅在服务器运行
        self.Box.OnComponentBeginOverlap:Add(self.BeginOverlap, self);
        self.Box.OnComponentEndOverlap:Add(self.EndOverlap, self);
    end

end

function MyActor:BeginOverlap()
    print("MyActor:BeginOverlap");
    self.MyEnumeration = MyEnum.Banana; -- 输出的值为2
    print(self.MyEnumeration);
end

function MyActor:EndOverlap()
    print("MyActor:EndOverlap");
    self.MyEnumeration = MyEnum.Orange; -- 输出的值为1
    print(self.MyEnumeration);
end
```

# 绿洲启元玩法开发框架介绍

在开发过程中，框架设计的重要性不言而喻，如果事先能对玩法开发的功能模块进行合理的规划和区分，善于利用引擎自带的框架设计，把他们放在引擎框架合理的位置中，可以省去大量设计框架和功能的时间，做到事半功倍，代码结构也可以更加清晰利于维护。

下面会介绍引擎中自带的框架设计都由哪些部分组成，以及他们的作用范围。 

<br>

## DS&Client 服务器与客户端

当一局玩法开始后，会生成一个DS（Dedicated Server），DS是所有数据的权威端，一般来说，几乎所有的Actor都是在DS进行创建，然后同步到Client，在Client生成一份复制品，设置为同步的参数，在DS修改后，也会跟着同步到Client对复制品Actor的参数进行同步修改。

所以玩法逻辑也分为DS逻辑和Client逻辑，从防作弊角度来说，只有DS的数据是可以信任的，Client的所有数据都有可能被内存修改篡改，所以大部分的数据都是依靠DS的数据作为执行依据， 玩法的流程逻辑，阶段数据，击杀逻辑，比分数据，胜利判断等都是在DS执行和管理，部分纯粹表现型的功能，比如UI，则是由DS数据同步至Client，Client执行相关UI逻辑。

<br>

## Actor场景中实体的基本类型

如果需要让某个实体对象在场景中生成，那么就必须继承自Actor类型（这里指的是继承链的最底层必须为Actor）。

Actor包含Location，Rotation，Scale等在场景中确定位置的属性。同时也有网络同步的相关职能，开发者可以通过绑定Lua脚本后，重载脚本中的BeginPlay（创建时触发），EndPlay（移除时触发），Tick（每帧触发）等基本事件来完成自己所需要的功能。

后续介绍的GameMode，GameState，PlayerController，PlayerPawn，PlayerState都是继承自Actor，所以具有以上说的Actor全部功能。

### **GameMode和GameState**

- GameMode 玩法模式模块

GameMode主要用于管理玩法流程，<span style="color:red">**实例仅在DS生成**</span>（在Client无法获取到GameMode），单局玩法中只存在一个实例。是可自定义的类型中，在DS最早生成的实例。

一般用于处理一些仅在DS存在和运行的数据和逻辑，玩法流程的编写可以自行在模式编辑器和GameMode中选择一样。

- GameState 数据模块

GameState主要用于管理玩法全局数据，和GameMode一样，在DS仅存在一份实例，但是每个Client都会存在一份DS的GameState的复制品实例，所以GameState是DS全局数据同步至Client的主要桥梁。

通过属性同步可以把DS的全局玩法数据同步至各个Client，如游戏阶段的倒计时，全局分数面板等常驻参数，即时信息则使用广播RPC广播至各个Client，如全局击杀播报，胜利播报等。

### **PlayerController、PlayerPawn和PlayerState**

- PlayerController 玩家控制器模块

PlayerController主要用于管理单个玩家自身控制相关功能。DS中每一位玩家都拥有一个各自的PlayerController实例，Client中则只有该玩家自身的PlayerController复制品实例，玩家的PlayerController在自己的Client无法获取。

PlayerController适合存放与玩家自身相关而其他玩家不需要知道的参数，如击杀数，分数，金币数等，同时玩家自身相关的功能逻辑也可以在此实现，如购买道具，仅自身可以收到的通知播报等。

同时PlayerController也是玩家Client和Server交互的主要途径，所有由Client需要告知Server的消息，都是由PlayerController进行传递，比如想实现一个购买商品的功能，那么应该是由玩家点击购买按钮，由UI通知PlayerController发送一个ServerRPC，向DS申请购买物品，DS里收到申请后再进行扣款，发放物品等操作。

- PlayerPawn（PlayerCharacter）玩家角色模块

PlayerPawn主要用于管理单个玩家所控制角色相关的功能，一般为玩家所操控的人形单位实体。DS中所有存在的角色实体都对应着一份PlayerPawn实例，Client中由于存在网络同步裁剪，只能获取到自身角色一定范围内的PlayerPawn复制品实例（距离自身太远的角色可能会被裁剪导致无法获取）。

PlayerPawn合适存放玩家角色相关参数，如护甲值，攻击力，血量等，角色强相关的逻辑也推荐在这里实现（如使用道具，使用技能，播放动画等）。

- PlayerState 玩家数据模块

PlayerState主要用于管理需要在队伍中同步的相关参数。在DS中，一个PlayerController实例对应一份PlayerState实例，在Client中则有你同队（TeamID相同）的所有玩家的PlayerState实例（Client无法获取到非队友玩家的PlayerState），用途相对来说局限性比较大。

PlayerState一般仅用于存放所有队友都需要知道，但是敌人不需要知道的参数，如队伍面板需要显示的生命值，需要在小地图显示的坐标位置等）。

<br>

## UI系统

UI是玩家和游戏交互最主要的途径，需要注意的一点是<span style="color:red">**UI仅在Client存在**</span>。

在绿洲启元编辑器的用户控件编辑界面中，开发者可以创建UI的基本布局，在绑定脚本进行UI逻辑的控制，通过Construct，Tick等基础事件开发需要的功能。一般UI仅对DS同步至客户端的展示型数据进行处理和展示，开始购买商品和使用技能等逻辑则是调用Controller的ServerRPC进行申请，UI自身不储存重要的数据和执行重要的逻辑。

<br>

## 网络同步系统

网络同步系统在需要联机的玩法开发中至关重要，只有了解并掌握了网络同步系统的相关功能，才能顺利的进行玩法的开发。网络同步系统具体的介绍和使用方式在 [网络同步系统介绍](https://developer.gp.qq.com/wikieditor/#/catalog/203) 中。

## 部分内容补充

### UID

UID即玩家唯一ID，是玩家在和平中唯一标识ID，类似账号，在整个和平中，每个账号的UID都是独一无二的，主要用于一些玩家数据管理用标识，如存档接口就需要传入玩家UID来进行存档和读档。

获取方法：PlayerState.UID，PlayerController.PlayerUID

### PlayerKey

PlayerKey即玩家Key，是玩家在单局玩法中临时生成的一个标识码，在单局玩法中，该局每个玩家的PlayerKey都是唯一的，是玩法内寻找对应玩家的重要参数，大部分单局玩法获取角色实例，获取角色数据相关功能，都需要用到PlayerKey。

获取方法：PlayerState.PlayerKey，PlayerController.PlayerKey，PlayerCharacter.PlayerKey

### TeamID

TeamID即玩家队伍ID，意味着你该局中你属于哪个队伍，该参数初始值会根据玩法上传时的匹配参数决定，后续也可以通过相关接口在游戏中二次分队，获取小队成员，判断敌我阵营，以及部分和平功能，如小地图队友标记，左上角的队友状态栏等功能，都和TeamID有关。

获取方法：PlayerState.TeamID，PlayerController.TeamID，PlayerCharacter.TeamID

# 自定义行为统计

开发者可以制定具体的数据指标来跟踪玩家的行为模式与偏好，玩法上线后通过分析这些数据，针对性优化游戏的设计与体验，包含但不限于：
- 玩家行为：某类道具的使用次数与频率如何？
- 关卡体验：关卡难度设计是否合理？玩家游玩进度情况？
- 交互设计：有多少玩家点击或触发过某按钮或功能？

绿洲启元编辑器通过自定义行为上报的API支持了特定数据的收集，并于绿洲启元管理平台提供了 [查询统计数据](https://gpeditor.qq.com/user#operationalData/2) 的途径。

## 案例演示

抽奖是游戏中较为常见的一类功能，玩家通过抽奖行为获得预设的相对应的奖励，开发者要验证抽奖概率的准确性就依赖有效的玩家数据作为基准进行分析。以跟踪和分析玩家在游戏中的抽奖行为为例，做一个简单的演示。

如下所示，设计了一个具有三种奖品的奖池，每当一个奖品被抽中时，就将该事件进行上报。

|抽奖奖品|中奖概率|中奖行为ID|
|-|-|-|
|AKM|40%|800800|
|M416|30%|800801|
|AUG|30%|800803|

---

**抽奖行为上报**

``` lua
function Lottery:LotteryHandle(PlayerKey)
    -- 判断PlayerKey以及PlayerKey的有效性
    if not PlayerKey or not UGCGameSystem.GetPlayerPawnByPlayerKey(PlayerKey) then return end

    -- 自定义一个奖品列表
    local rewardInfo = {
        result001 = {
            resultId = "AKM",
            eventId = 800801,
            rate = 40
        },
        result002 = {
            resultId = "M416",
            eventId = 800802,
            rate = 30
        },
        result003 = {
            resultId = "AUG",
            eventId = 800803,
            rate = 30
        },
    }

    -- 预创建抽奖结果，随机数
    local result = rewardInfo.result001
    local rate = math.random(1,100)

    local addNum = 0
    for k,v in pairs(rewardInfo) do
        if rate <= addNum + v.rate then
            -- 真实抽奖结果赋值
            result = v
            break
                                else
                                                addNum = addNum + v.rate
        end
    end

    -- 上报抽奖结果事件
    UGCGameSystem.SendGreyTLog(result.eventId, PlayerKey)

end
```

在这段代码逻辑中，我们设计了一个简单的抽奖功能，在处理抽奖结果中上报了该行为事件。

如上所示，当抽奖结果result返回后，使用 ``UGCGameSystem.SendGreyTLog`` 将结果进行上报，其中 ``eventId`` 对应的是每次抽奖结果的事件，一共有三个抽奖奖品，对应的事件ID分别为：800800，800801，800803。

> **注意：**
行为上报只作为收集数据的方式，不应当对游戏流程产生任何的阻塞，并破坏玩家的游戏体验。

登录 [管理平台](https://gpeditor.qq.com/user#operationalData/2)，选择 ``运营数据-自定义埋点数据`` tab页，选择对应的游戏项目，并输入 ``800800、800801与800803``，查询这三个ID所对应的奖品在 ``最近30天`` 的中奖情况：

# 玩家数据存档

数据存档功能允许开发者将对局中产生的玩家个人数据跨对局、持久化存储，例如玩家获得的装备物品、历史战绩等等，各玩法之间的存档数据相互隔离，因此开发者可以安全地使用存档数据。

<br>

## 存档数据结构

存档数据的类型和结构为 [LuaTable](https://www.runoob.com/lua/lua-tables.html)，table中的数据元素支持复合类型，包括数字、字符串等基础数据类型，也可以为数组、字典等复杂嵌套结构数据。

> 单玩家的存档数据容量大小不能超过256Kb

```lua
SaveData = {}

-- 基础类型元素
SaveData[1] = "Lua"
SaveData["wow"] = "abc"

-- table类型元素
local SubTableData = {}
table.insert(SubTableData,"equip")
table.insert(SubTableData,"avatar")
SaveData["attributeList"] = SubTableData
```

<br>

## 保存存档

使用 [SavePlayerArchiveData](https://developer.gp.qq.com/api/#/searchContent/UGCPlayerStateSystem?classDetailShow=true&path=class%2Fdetail%2F%E5%92%8C%E5%B9%B3%E5%85%A8%E5%B1%80%E6%8E%A5%E5%8F%A3%2F%E8%A7%92%E8%89%B2%E7%B3%BB%E7%BB%9F%2FUGCPlayerStateSystem.json&isSelect=1&apiEnc=%5B%22%E7%B1%BB%22%5D&apiLabel=UGCPlayerStateSystem&autoJump=SavePlayerArchiveData) 接口存储玩家的存档数据，通过参数 [UID](https://developer.gp.qq.com/wikieditor/#/catalog/352) 标识数据的拥有者。

```lua   
local equipSequence = {}

local playerPawn = self:GetPlayerCharacterSafety()
local allItemData = UGCBackPackSystem.GetAllItemData(playerPawn)
for key, itemData in pairs(allItemData) do
	table.insert(equipSequence, itemData.ItemID)
end

local playerUID = UGCPawnAttrSystem.GetPlayerUID(playerPawn)
UGCPlayerStateSystem.SavePlayerArchiveData(tonumber(playerUID), equipSequence)
```

<br>

## 获取存档

调用 [GetPlayerArchiveData](https://developer.gp.qq.com/api/#/searchContent/UGCPlayerStateSystem?classDetailShow=true&path=class%2Fdetail%2F%E5%92%8C%E5%B9%B3%E5%85%A8%E5%B1%80%E6%8E%A5%E5%8F%A3%2F%E8%A7%92%E8%89%B2%E7%B3%BB%E7%BB%9F%2FUGCPlayerStateSystem.json&isSelect=1&apiEnc=%5B%22%E7%B1%BB%22%5D&apiLabel=UGCPlayerStateSystem&autoJump=GetPlayerArchiveData) 接口以获取指定玩家的存档数据。
	
```lua
local playerPawn = self:GetPlayerCharacterSafety()
	
local playerUID = UGCPawnAttrSystem.GetPlayerUID(playerPawn)
local readArchive = UGCPlayerStateSystem.GetPlayerArchiveData(tonumber(playerUID))
for i = 1, #readArchive do
	UGCBackPackSystem.AddItem(playerPawn, readArchive[i], 1)
end
```

## 存档调试

存档功能已支持编辑器内 [Debug调试](https://developer.gp.qq.com/wikieditor/?timeStamp=1725589224096#/catalog/307) 与 [移动端测试](https://developer.gp.qq.com/wikieditor/?timeStamp=1729765192241#/catalog/372)，其中Debug调试环境下支持跨DS存储，但仅限一次调试会话内有效。

例如，接入了 [二次匹配](https://developer.gp.qq.com/wikieditor/?timeStamp=1725589224096#/catalog/376) 功能的玩法在一次调试会话内，不同子模式（即不同DS）之间的存档数据可互通，但重启调试后存档数据无法再被读取。

> Debug调试环境下每场对局存储的数据非持久化，每次重新启动调试均会重置数据

<br>

## 注意事项

开发者在使用存档功能时需要特别注意以下事项：

- 请勿在 [发送游戏结算数据](https://developer.gp.qq.com/wikieditor/#/catalog/272) 之后调用SavePlayerArchiveData，否则会出现数据丢档的问题
- 存档数据针对玩家个人，请勿把全局玩家的数据都存储在同一个UID下
- 存档功能支持数据的读写，但不支持删除操作，如果游戏业务中确定需要清除玩家的数据，可以将存档数据置nil，历史态数据无法追溯

## 粒子模块介绍

特效编辑器中，每个发射器包含多个不同类型的粒子模块，可通过启用并修改粒子模块来改变发射器的表现效果。

### 基础设置

基础设置模块用于控制粒子发射的基本行为。

![image.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/tHQU2image.png)

|属性名称|说明|
|:-:|-|
|发射率|设置发射器在播放期间每秒产生的粒子数量|
|爆发数量|设置单次爆发事件中瞬时生成的粒子数量|
|爆发次数|设置发射器单次播放周期内触发的爆发事件总数，设置为0时发射器会循环播放|
|持续时间|设置发射器单次播放的总时长，单位：秒|

---

### 材质

材质模块用于定义粒子渲染所使用的材质资源及其动画属性。

![image.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/DKwZzimage.png)

![UV数量对比.gif](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/HaQuHUV%E6%95%B0%E9%87%8F%E5%AF%B9%E6%AF%94.gif)

|属性名称|说明|
|:-:|-|
|发射器材质|指定用于渲染粒子外观的材质资源|
|发射顺序|控制粒子在生命周期内，如何从UV划分出的子图格序列中选择帧进行播放<br>• 无：不进行动画播放，每个粒子始终显示材质的原始完整图片<br>• 顺序：粒子按划分后的子图格，严格遵循从左到右、从上到下的顺序逐帧播放，形成连续动画<br>• 随机：粒子在播放时，从所有子图格中随机选择一帧显示，适用于产生变化丰富的非连续效果|
|UV数量|此参数将材质纹理在水平(U)和垂直(V)方向上划分为一个网格，子图格编号按从左到右、从上到下的顺序排列|
|起始|设置UV动画播放序列的起始帧编号，【发射顺序】选择“顺序”时生效|
|结束|设置UV动画播放序列的结束帧编号，【发射顺序】选择“顺序”时生效|
|对齐方式|材质的对齐方式会影响粒子大小的控制<br>• 正方形：粒子尺寸由大小相关模块的X值控制<br>![image.36.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/0rKskimage.36.png)<br>• 长方形：粒子尺寸由大小相关模块的X值和Y值控制<br>![image.35.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/SUSP1image.35.png)<br>• 速度：粒子尺寸受 速度在X/Y轴的分量影响（速度越快，粒子在对应方向的尺寸越大）|

### 摄像机偏移

摄像机偏移用于微调粒子与摄像机之间相对位置，填入的数值越大，粒子离摄像机越近。通常用于以下两种情况：
- 避免摄像机穿透：当摄像机靠近或移动通过粒子效果（如雾气、灰尘）时，过近的距离会导致粒子不自然地放大或出现视觉瑕疵。通过设置适当的 Z 轴正偏移值（例如设置为 50，表示粒子沿摄像机前方偏移 50 个单位），可以确保粒子始终与摄像机保持一定距离，维持视觉上的合理性。
- 创建UI关联特效：对于需要固定在屏幕特定位置的粒子元素（如角色状态图标、对话气泡等），可通过计算并设置合适的偏移量，使粒子效果“附着”于屏幕指定区域，不受摄像机移动或旋转的影响，实现 UI 级别的视觉表现。

![image.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/ljJIximage.png)

|属性名称|说明|
|:-:|-|
|类型|设置摄像机偏移的方式<br>• 常数：粒子的摄像机偏移值恒定不变<br>• 曲线：粒子的摄像机偏移值在其生命周期内，从开始值平滑过渡到结束值|
|值|设定粒子固定的摄像机偏移值，【类型】选择“常数”时生效|
|开始|粒子在生命周期初始时刻的摄像机偏移值，【类型】选择“曲线”时生效|
|结束|粒子在生命周期结束时刻的摄像机偏移值，【类型】选择“曲线”时生效|

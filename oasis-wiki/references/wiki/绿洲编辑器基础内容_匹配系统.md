# 匹配设置

匹配是指将和平精英大厅中不同的组队/个人凑成战斗所需要的队伍数和人数的过程，在绿洲启元编辑器中，匹配无须开发者写代码，只需要简单的配置即可实现匹配。

## 配置说明

---

### 1. 详细介绍

#### 1.1 队伍设置

- 【队伍数量】：指需要大厅丢到战斗的队伍数
- 【小队玩家数量】：指每个队伍的玩家数
- 【注意】：队伍数量是指大厅丢给战斗的队伍数，并不意味着一定等于战斗内的实际分队情况。例如：《小红帽与大灰狼》玩法中配置的队伍数是1，小队玩家数量是9，那么大厅在匹配时会将9名玩家丢给战斗，之后，《小红帽与大灰狼》工程中再对这9人重新分队，分成6小红帽vs3大灰狼

#### 1.2 匹配方法

- 【匹配方法】：
  1. PeaceClassic: 用和平经典段位分匹配，适用于类经典模式或者钢枪型玩法
  2. Random: 完全随机匹配，适用于纯娱乐型玩法
  3. UserDefined：用户自定义分数匹配，适用于玩法会有水平差异，但又不适合用经典模式的段位分（例如小红帽与大灰狼玩法）
- 【初始分数】：仅当选择UserDefined时生效，指玩家初始的匹配分，定死2000
- 【最低分数】：仅当选择UserDefined时生效，指玩家最低的匹配分，定死0
- 【最高分数】：仅当选择UserDefined时生效，指玩家最高的匹配分，定死4000
- 【匹配小段分数】：每100分为一个匹配分段（类似和平，青铜5到青铜4为100分）
- 【匹配扩段时间】：默认10，指10秒内只在同分数段内的人匹配，超过10秒扩到+-1段，再超过10秒扩到+-2段，以此类推
- 【匹配扩段数量】：默认10，指最多扩到+-10段
- 【组队匹配分计算】：
  1. AverageScore:指组队的人按组队的平均分计算匹配分
  2. HighestScore:指组队的人按组队中的最高分计算匹配分

#### 1.3 匹配分lua接口（仅匹配方法选择UserDefined需要）

<details open>
<summary> <font face="楷体">示例：关键接口</font> </summary>

```lua
---生效范围: S                                                      
    PlayerState.UGCPlayerInfo.MatchRank --获得每个玩家的PlayerState之后可用此变量读写玩家的自定义匹配分
```

## 二次匹配功能

> 文档路径: 绿洲编辑器基础内容 > 匹配系统 > 二次匹配功能

**涉及API:** `ApplyPlayerJoinLimitCount`, `GetTeamLeaderKeyByTeamID`, `MatchSetting`, `NotifyRequestToJoinLobbyTeamDelegate`, `RequestCancelMatch`, `RequestMatch`, `RequestReadyMatch`, `RequestToJoinLobbyTeam`, `RespondToRequestingToJoinLobbyTeam`, `UGCMultiMode`, `UGCMultiMode.RequestCancelMatch`

# 二次匹配功能

绿洲启元编辑器目前已支持在玩法中设置多个子模式进行跨DS匹配，且能够在不返回和平大厅的情况下继续发起新的子模式匹配，该机制可称作为二次匹配，二次匹配功能具有以下特点：

- 同一玩法下可配置多个子模式（不建议超过6个），通常具有一个默认子模式
- 每个子模式可以单独设置匹配队伍人数，单独设置成局条件
- 在子模式内可以发起另一个子模式的匹配，无需返回大厅，每个子模式为独立的DS进程

<br>

## 多模式配置

### 启用多模式

玩法工程想支持多模式，只需在 ``编辑->工程设置->Match Setting`` 中进行设置即可。

![image.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/GXjquimage.png)

Match Setting中增加了 ``Multi-mode Game`` 配置，玩法工程默认启用 ``Single Mode Game`` 即单模式配置；想要开启多模式，需要将 ``Multi-mode Game`` 中的“使用多模式配置”勾选上，此时单模式配置将自动失效。

![image.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/jLfwlimage.png)

<br>

### 匹配设置

在多模式配置中，可以根据玩法需要添加子模式配置，但目前不建议配置过多子模式，建议在4个最佳，不建议超过6个。

![image.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/KxCMtimage.png)

每个子模式的配置项相同：

![image.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/Tsbqrimage.png)

|配置项|说明|
|-|-|
|设为默认模式|该模式是否作为进入玩法的默认进入模式|
|模式ID|玩法内该子模式唯一ID，自动分配|
|模式名称|该子模式的名称，将用于玩法内的子模式名称显示|
|模式图片|该子模式的缩略图。将用于玩法内模式选择时显示|
|模式解锁条件描述|解锁子模式的条件描述|

<br>

**注意事项**
> 1. 多模式配置下必须要有一个子模式作为默认模式，默认勾选第一个子模式，可根据玩法实际需求选择其他子模式作为默认模式
> 2. 模式解锁不会作为默认支持功能，需要玩法中其他逻辑配套支持，该配置仅作为子模式是否可选的展示功能

<br>

子模式配置下的 ``MatchSetting`` 与单模式配置相同，主要用于配置队伍数、队伍人数等匹配相关参数，可以前往 [匹配设置](https://developer.gp.qq.com/wikieditor/#/catalog/127) 查看匹配相关详细配置。

<br>

## 多模式匹配

绿洲启元编辑器内提供的可视化配置仅能调整子模式显示和匹配相关设置，更多细致化的功能，如解锁模式、模式补人等，需要使用脚本功能实现，二次匹配功能提供了一系列相关API：

**UGCMultiMode**

|API|描述|<div style="width:120px;">生效范围|<div style="width:250px;">参数|返回值|
|--|--|--|--|--|
|SetModeChooseUIVisible|设置模式选择用户界面（UI）的显示/隐藏状态|客户端|Visible：显示/隐藏 ||
|SetModeState|设置指定模式的可选择状态|客户端|ModeID：设置目标模式ID<br> ModeAvailability：可用/不可用||True/False|
|GetModeID|获取当前活跃的模式ID|客户端/服务端||当前模式ID，如不存在活跃模式则返回0|
|SetModeChooseButtonVisible|设置模式选择按钮的显示/隐藏状态|客户端|Visible：显示/隐藏|True/False|
|SetPlayerFill|开启或关闭补人功能|服务端|bPlayerFill：开启/关闭||
|RequestMatch|发起指定模式匹配|客户端|SubModeID：匹配目标子模式ID<br>ResCallBack：结果回调<br>obj：上下文实例||
|RequestCancelMatch|取消匹配|客户端||

开启模式选择UI
``` lua
function UGCGameState:ReceiveBeginPlay()
    if not UGCGameSystem.IsServer() then
        UGCMultiMode.SetModeChooseUIVisible(true)
        print(self.GameModeID)
    end
end
```

发起指定子模式的匹配
``` lua
--定义匹配结果回调函数
function TestButton_UIBP:RequestMatchRes(result)
    if result then
        PopUpNoticeUI.ShowNewNoticeQueue("TestButton_UIBP:RequestMatchRes success!")
    else
        PopUpNoticeUI.ShowNewNoticeQueue("TestButton_UIBP:RequestMatchRes fail!")
    end
end

--1001模式匹配
function TestButton_UIBP:RequestMatch1001()
    UGCMultiMode.RequestMatch(1001, self.RequestMatchRes, self)
end

--1002模式匹配
function TestButton_UIBP:RequestMatch1002()
    UGCMultiMode.RequestMatch(1002, self.RequestMatchRes, self)
end
```

取消匹配
```lua
function TestButton_UIBP:RequestCancelMatch()
    UGCMultiMode.RequestCancelMatch()
end
```

<br>

>注意：```RequestMatch``` 和 ```RequestCancelMatch``` 不能在同一帧内调用，有1秒的限频调用限制。

## 多模式调试

进入玩法后，可以查看子模式列表（列表显示配置中填写的子模式名称和缩略图）

**注意事项**
	
[Debug调试](https://developer.gp.qq.com/wikieditor/#/catalog/307) 下的二次匹配功能与移动端略有不同，具有以下特殊规则：
	
- 对局内的任意玩家均可发起二次匹配，且匹配结果必定成功，匹配过程中无法主动取消
- 二次匹配成功后，当前对局的DS会将所有的玩家（包括非队友玩家）全部转移到新的对局中，并关闭旧的对局DS

## 大厅队伍功能扩展

### 加入大厅队伍
	
子模式内的玩家允许加入指定的大厅队伍，一旦成功加入，该玩家将具备与其他成员的大厅队伍关系，不同于玩法内的临时队伍关系，玩家返回大厅将保留队伍状态，开发者可以基于 [``RequestToJoinLobbyTeam``](https://developer.gp.qq.com/api/#/searchContent/UGCTeamSystem?classDetailShow=true&path=class%2Fdetail%2F%E5%92%8C%E5%B9%B3%E5%85%A8%E5%B1%80%E6%8E%A5%E5%8F%A3%2F%E7%A4%BE%E4%BA%A4%E7%B3%BB%E7%BB%9F%2FUGCTeamSystem.json&isSelect=1&apiEnc=%5B%22%E7%B1%BB%22%5D&apiLabel=UGCTeamSystem&autoJump=RequestToJoinLobbyTeam)、[``NotifyRequestToJoinLobbyTeamDelegate``](https://developer.gp.qq.com/api/#/searchContent/UGCTeamSystem?classDetailShow=true&path=class%2Fdetail%2F%E5%92%8C%E5%B9%B3%E5%85%A8%E5%B1%80%E6%8E%A5%E5%8F%A3%2F%E7%A4%BE%E4%BA%A4%E7%B3%BB%E7%BB%9F%2FUGCTeamSystem.json&isSelect=1&apiEnc=%5B%22%E7%B1%BB%22%5D&apiLabel=UGCTeamSystem)、[``RespondToRequestingToJoinLobbyTeam``](https://developer.gp.qq.com/api/#/searchContent/UGCTeamSystem?classDetailShow=true&path=class%2Fdetail%2F%E5%92%8C%E5%B9%B3%E5%85%A8%E5%B1%80%E6%8E%A5%E5%8F%A3%2F%E7%A4%BE%E4%BA%A4%E7%B3%BB%E7%BB%9F%2FUGCTeamSystem.json&isSelect=1&apiEnc=%5B%22%E7%B1%BB%22%5D&apiLabel=UGCTeamSystem&autoJump=RespondToRequestingToJoinLobbyTeam) 等API实现从申请加入队伍到响应加入的完整交互功能。
	
> 该功能仅在启用了二次匹配的玩法中生效
	
---	
	
### 同进同出子模式

具备大厅队伍关系的玩家将持久保留组队关系，全队同进同出各子模式，哪怕玩家中途离开对局返回大厅，大厅内的队伍关系依然存在。
	
![3de0386eebb68c89e25e6f04aa68ff9b.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/si8If3de0386eebb68c89e25e6f04aa68ff9b.png)
	
子模式内当队员均处于准备状态时，可由队长发起二次匹配，匹配状态基于队长进行同步，同队玩家一起进入新的子模式DS，开发者可以结合 [``RequestReadyMatch``](https://developer.gp.qq.com/api/#/searchContent/UGCMultiMode?classDetailShow=true&path=class%2Fdetail%2F%E5%92%8C%E5%B9%B3%E5%85%A8%E5%B1%80%E6%8E%A5%E5%8F%A3%2F%E7%8E%A9%E6%B3%95%E8%A7%84%E5%88%99%2FUGCMultiMode.json&isSelect=1&apiEnc=%5B%22%E7%B1%BB%22%5D&apiLabel=UGCMultiMode&autoJump=RequestReadyMatch) 和 [``GetTeamLeaderKeyByTeamID``](https://developer.gp.qq.com/api/#/searchContent/UGCTeamSystem?classDetailShow=true&path=class%2Fdetail%2F%E5%92%8C%E5%B9%B3%E5%85%A8%E5%B1%80%E6%8E%A5%E5%8F%A3%2F%E7%A4%BE%E4%BA%A4%E7%B3%BB%E7%BB%9F%2FUGCTeamSystem.json&isSelect=1&apiEnc=%5B%22%E7%B1%BB%22%5D&apiLabel=UGCTeamSystem&autoJump=GetTeamLeaderKeyByTeamID) 等API自行实现相关交互效果。
	
子模式中支持补人机制，当队伍内玩家人数不足以进入战斗阶段时，可以调用 [``ApplyPlayerJoinLimitCount``](https://developer.gp.qq.com/api/#/searchContent/UGCGameSystem?classDetailShow=true&path=class%2Fdetail%2F%E5%92%8C%E5%B9%B3%E5%85%A8%E5%B1%80%E6%8E%A5%E5%8F%A3%2F%E5%9F%BA%E7%A1%80%E5%8A%9F%E8%83%BD%2FUGCGameSystem.json&isSelect=1&apiEnc=%5B%22%E7%B1%BB%22%5D&apiLabel=UGCGameSystem&autoJump=ApplyPlayerJoinLimitCount) 补充新玩家，但此方式添加的玩家基于 [玩法内队伍关系](https://developer.gp.qq.com/wikieditor/#/catalog/20118?autoJump=%E9%98%9F%E4%BC%8D%E7%B3%BB%E7%BB%9F)，非大厅队伍关系，因此该玩家无法与大厅队伍成员一起进入其他子模式，且返回和平大厅后队伍内也不包含该玩家。

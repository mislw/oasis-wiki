## 需求背景

防御塔，常见于MOBA游戏和塔防游戏中，一般来说，即是一个具有一定攻击范围且无法移动的实体，当敌对阵营的目标出现在塔的攻击范围内时，防御塔则会以一定的攻击频率不断释放自身技能攻击敌人。

[图片#4668]

## 实现要点

一：具有实体，可被攻击、摧毁，具有生命值

二：具有完善的阵营判断机制，可以筛选出敌对目标进行攻击

三：具有完善的仇恨索敌机制，可以选取到攻击范围内的目标

四：具有简易的行为逻辑控制，在有目标时，需要以一定间隔不断释放自身技能

五：攻击范围的指示器绘制

六：防御塔技能的实现（可自行横向扩展）

## 制作流程

下面我们将以一个具体的防御塔需求来讲解，如何实现此类防御塔的通用需求。射线塔：攻击范围内若有目标则会发射射线对目标造成持续伤害，若目标和防御塔之间有碰撞体，则射线会被碰撞体阻挡。

## 第五部分：Q & A

### FAQ — 开发中最常被问到的问题

**Q: 创建的物品在游戏里不显示怎么办？**

A:检查是否在场景中放置了 UGCPickupWrapper_BP 并填入了正确的物品 ID

确认物品的「拾取物模型」不为空、物品表对应ID的Handle是否为空、物品ID是否存在

**Q: 配件为什么装不到枪上？**

A:检查配件的「物品标记 (Tag)」是否匹配枪械槽位的 AllowTags

检查配件的「自定义类型」是否匹配枪械槽位的 AllowTypes

匹配规则：AllowTypes 或 AllowTags 任一匹配即可

**Q: 创建的物品在背包里看不到使用/丢弃按钮？**

A:检查 V2 背包配置中的「是否可使用」「是否可丢弃」「是否可销毁」选项

检查是否启用了 V2 背包（未启用则走经典背包逻辑）

丢弃、超限的默认处理方式：生成地面拾取物

**Q: 怎么让物品跨局保留？**

A:勾选「是否持久化」

物品需要存入仓库才能跨局保留（战斗背包的物品局内有效）

框架自动处理 45 秒定时 + 结算前强制保存

**Q: 自定义模型导入后位置不对？**

A:检查 DCC 软件中的单位设置（需用厘米）

检查 Root 位置是否放在角色手持位置

使用 GIZMO 调整挂接偏移

确认枪口朝向 X 轴正方向

**Q: 为什么我的枪械开火没有伤害？**

A:检查枪械 WeaponEntity 的「伤害数值」配置

如果用了技能枪械，确认技能中的伤害轨道配置正确

**Q：枪械自定义配件的改装配置，是否支持跨对局保存呢？**

A：是的，枪械自定义配件的改装配置支持跨对局保存。 玩家在对局中进行的配件改装配置会保留至后续对局，无需每次重新设置。

**Q：保存 save 字段后，客户端有回调吗？还是只能在客户端延迟查询？**

A：不需要额外延迟查询。 save 字段通过 ds 调用，服务端会自动同步回客户端，客户端可直接接收同步结果。UUGCBattleItemInstanceDataManager::OnBattleItemInstanceDataChange委托可以客户端监听变化。

**Q：枪的外观，在哪里修改?是在蓝图里面吗?**

[图片#4665]

A：根据修改需求的不同，分两种情况：

1、只需替换整枪模型（裸枪）

直接在 Weapon Handle 上修改模型配置即可。Weapon Handle 即枪械物品的模型配置入口，将整枪模型替换到此处即可。

2、需要按配件逐件组装（枪口、握把等零散部件）

由于配件模型通常是零散的，需要单独创建一个武器，进入该武器的换皮配置中进行逐部件修改。

补充说明：

● 如果你的枪本身就是一整个完整模型（已包含枪口、握把等），直接在 Handle 上替换即可。

● 修改后配置会自动持久化，无需额外保存。

**Q:一个实例化的物品ID，分子ID，子I D再对应对应蓝图的外观吗?**

A：理论上可以。 实例化物品可以用一张数据表来配置，你可以在表中为不同实例配置所需的 mesh（外观模型）。

举例说明：

● 第 1 个实例（子 ID 1）→ 对应 Mesh A

● 第 2 个实例（子 ID 2）→ 对应 Mesh B

注意事项：

需自行实现挂载逻辑：不同 mesh 对应的外观虽然可以在表中配置，但"如何将该 mesh 装备到角色手上"的实现需要自行完成。
实例数据建议精简存储：物品的实例数据建议只存一个关键字段，后续数据通过关联读取获取，不要将整行数据全部写入实例数据中，以避免数据冗余和维护成本。

**Q：手持武器的动画能改吗？**

A：目前暂不支持修改手持武器动画。

原因： 武器动画数量庞大，涉及走、站、蹲、趴等多种状态，一套完整的武器动画可能包含 70～80 个动画片段，目前这些片段尚未对外开放修改。

后续计划： 后续会评估是否有方案将该能力开放出来，请关注版本更新公告。

**Q：空白消耗物建了后不能直接使用，具体怎么用啊？**

A：空白消耗品本身不含任何使用逻辑，所有行为均需自行实现。

为什么不能直接用？

"空白"即代表全部为空——创建一个空白消耗品后，它不会自带任何功能。点击"使用"时不会自动触发效果、加 Buff 或执行扣除，一切都需要开发者自行编写逻辑。

需要自行实现的内容：

使用判定：判断该消耗品当前是否可以使用。
BeginUse（开始使用）：监听开始使用事件，编写对应逻辑。
StopUse（结束使用）：监听结束使用事件，编写对应逻辑。
效果触发：例如给角色添加 Buff、触发技能等，需自行处理。
消耗扣除：使用成功后，执行物品数量扣除，需自行完成。

小结： 空白消耗品仅提供数据框架，使用逻辑、效果触发、扣除逻辑均需开发者自行实现，系统不会自动处理。

**Q：物品ID，不是只能对应一个蓝图class吗，现在可以多个了吗？**

A：可以。 目前的实现方式为：一个物品 ID 对应一个蓝图，同时配合一张额外的配置表，通过一个配置 ID 即可生成多个 Define ID 的效果。

简要说明：

● 物品 ID ↔ 蓝图：仍然保持一对一关系。

● 配置表：新增的配置表用于扩展，通过配置 ID 映射出多个 Define ID，实现"一物多态"的效果。

**Q：物品的 UI 需要自己制作吗？**

A：目前无需自行制作 UI。 只需复写（Override）物品的属性配置，即可直接修改对应显示效果，不需要单独开发 UI。

后续规划： 后续将开放背包 UI，届时可支持更多 UI 层面的自定义效果，以及对物品建模的修改。

**Q:一个枪械蓝图，可以对应多个物品ID吗，就是多个ID共用一个蓝图。**

A：可以。 一个枪械蓝图支持对应多个物品 ID。

● 例如编号 8310 的枪械，其中子编号 1～5 均可共用同一个枪械蓝图。

开发者们好，本文主要通过参考玩家反馈、绿洲玩法常见问题，帮助开发者从玩法设计层面规避部分基础设计问题。

开发者在玩法、游戏内活动制作中，可参考本文中的规范与建议进行调整。

**一、玩法基础设计**

开局准备时间

过长的对局前准备时间会给玩家带来负面的游戏体验，根据玩法本身的需求，设计合理的准备时间有助于提升玩家体验。

建议时长：10-15S

其他优化：

对战类玩法，对局中所有玩家完成准备后，可提前开始对局，并制作提示信息
设计有复杂配置的玩法，可制作“常用配置”“保存配置”并能一键启用的功能
增加对局等待时间可休闲内容、游戏教程提示类内容

**玩家PVP类玩法复活无敌时间**

射击类游戏中需要在玩家“复活”后设计一定的无敌时间，避免玩家还未加载完成就被击杀，或复活后直接处于劣势。

建议设计：1-2S的无敌时间，无敌状态下开火后即刻接触无敌状态

**玩法初始武器配置**

发放给玩家的武器射击手感和玩家体验息息相关，由于大部分玩家使用移动设备进行游戏，单发/连发类武器的射击体验一般较差。

**建议初始发放的枪械和瞄具：**

手枪：P18C

步枪：M416

瞄具：红点、四倍镜

**不建议选择枪械（举例）：**

左轮：换弹时间长，无法连发，双指玩家射击体验感受差

M16A4：非连发枪械，双指玩家射击体验感受差

S686：弹匣容量少，PVE玩法中使用体验差

**常见按键冲突问题**

和平内玩家往往采用双指、多指操作，为保障玩家在操作层面的游戏体验，建议开发者接入【自定义按键功能】并在基础的界面设计中规避下述操作区域：

**二、玩法活动基础设计**

**玩法活动时间常见设计问题**

活动时间的设置与展示需要足够清晰，给予玩家明确的信息传达，从而避免理解错误可能导致的投诉与玩家体验下降。

**1、活动开启时间或结束于0点时，展示文本例：**

**建议设计：**

活动开始时间：4月11日0点

活动结束时间：4月22日23:59:59、4月22日23时59分59秒

**不建议设计：**

结束时间：4月23日0:00:00、4月22日24时（该类时间显示容易造成玩家对活动结束时间的误解）

**2、复杂活动设计合理性自查**

**重点提醒：**

复杂活动建议开发者自查参与逻辑上是否存在不合理设计，避免存在不同时间点开放内容设计导致的无法玩法任务、任务进度异常等情况：

异常情况，例：玩法开启时间为4月22日0点，玩家可开始体验活动内容，但活动内任务系统于当日10点开启，玩家此前进行的活动进度未正常计入任务重，导致出现客诉与玩家不满的情况。

**3、特殊时间不设计活动的开启与风险自查**

**重点提醒：**

针对闰年2月底时间附近的活动上线，需要额外进行跨2月29日的测试与逻辑自查，避免活动的时间逻辑存在错误导致异常情况出现。

玩法活动内容冗余设计建议

通常来说为保障玩家的游戏体验，玩法活动内奖励的获取要求要远低于“玩家全勤下可进行的游戏进度”，从而保障玩家的游戏体验

**建议设计：**

例通行证类活动中：获得全部奖励的理想时间设置为活动总时长的70%以下，重要、限定、高价值物品可确保玩家在活动总时长的30%以下可获得。

**三、玩法内界面设计的普适性体验优化方向**

**游戏界面的基础建议**

通常来说，游戏界面的操作逻辑已是不成文的规定，无论是玩家（甚至可以说是电子产品使用者）还是游戏开发商均会遵守一类逻辑，因此在界面的交互方式设计时，非常不建议打破常规，往往可能影响玩家的实际体验。

**界面关闭设计：**

1、遵循和平界面的关闭逻辑，设置在右上角关闭按钮，明显且便于点击

[图片#4319]

**需注意：活动界面设计时需要注意其层级，避免玩家的游戏操作按钮层级在其之上，导致玩家无法正常关闭页面**

2、关闭icon设计，建议：直接使用“X”形符号或使用文字“关闭”

3、多开页面的关闭逻辑，建议：遵守“一开一关”的逻辑

**不建议的界面关闭设计**

● 点击原界面图标进行页面关闭、关闭按钮设置在左上角（模拟器玩家集中在win端，不适用macOS的操作逻辑）

● 玩法内如果可能出现多个界面重叠，设计了多种不同的关闭逻辑

● 关闭按钮过度设计，过于华丽导致辨识性不强

● 当某主页面可开启子页面时，子页面的关闭不得将主页面一并关闭

Q&A

**1、问题：2.5实操项目的工程可以发下吗？**

回答：可以联系小助手，小助手会将讲解的工程文件发给大家。

**2、问题：只有2.5D模版建的项目才行吗？用game part可以吗？**

回答：为了赛事开发的效率和稳定性，请优先选择2.5D模板启动新项目。不是必须用2.5D模板，但强烈推荐使用。用普通项目只改Game Part是行不通的。

**3、问题：俯视角模板不是很适配 PC 模拟器，操作方式是不是只能做移动端？**

回答：操作方式不限于移动端，通过“输入映射”系统，可以完美适配PC端。原理：游戏内的操作（如移动、攻击、释放技能）并不直接绑定到某个具体的物理按键（如屏幕虚拟摇杆或键盘WASD），而是绑定到一个抽象的“逻辑输入”（如“移动方向”、“主攻击”）。

**4、问题：群体怪物伤害，现在怪物多了，容易掉线，没有办法做那种海岸生存。玩法如果怪带骨骼的大概能够支持到的量级是多少？**

回答：当前带骨骼怪物的性能上限约为50-100个。带骨骼动画的怪物数量建议控制在50个左右，可以保持流畅。官方正在积极优化，管理怪物数量以确保游戏流畅。

**5、问题：编辑器打开很慢，什么时候能优化？**

回答：编辑器性能优化是一个持续进行、长期推进的工程，官方已将其纳入常规迭代计划，暂时无法给出具体的完成时间表。

**6、问题：相机调成正交视角现在可以用吗？**

回答：正交视角功能暂不可用，但已经在做了。建议您优先尝试在现有透视相机框架下进行开发与调整，并密切关注官方更新。

**7、问题：3d 项目要改成2.5d 需要做哪些工作？**

回答：需要完整替换为2.5D（俯视角）模板的全套系统，不能只修改局部。工作量与风险较大，请谨慎评估。

具体需要更换的核心工作 ：

1、替换核心控制器与逻辑

● Game Part：这是游戏对象的核心逻辑组件，3D与2.5D的移动、碰撞、交互逻辑完全不同，必须替换。

● Player Control（角色控制器）：3D通常使用基于物理或字符控制器的移动，而2.5D俯视角通常采用平面（X-Z轴或X-Y轴）移动逻辑，并可能涉及与点击/摇杆操作的映射，需要完全重写。

2、重建输入映射系统

● 2.5D项目通常有专门的输入映射方案，以同时适配移动端的虚拟摇杆和PC端的键鼠/手柄。您需要将现有的3D输入系统替换为2.5D模板的输入系统。

3、调整或重制美术资源

● 角色与场景：3D模型和贴图可能需要调整，以适应俯视角下的视觉效果（例如，顶部视角下可能需要强化顶部细节）。

● 动画系统：骨骼动画可能需要调整或重做，以适配新的移动方向和平面动作逻辑。

● UI与特效：UI布局和世界空间特效需要根据新的屏幕空间和视角进行适配。

4、重构相机系统

● 需要将3D透视相机替换为固定角度（通常是45度）的俯视角相机。如果未来官方支持正交投影，还需进行相应设置。

5、调整物理与碰撞

● 物理交互可能需要从3D体积碰撞调整为更适合2.5D的平面碰撞或2D物理（取决于实现方式）。

**8、问题：摄像机怎么旋转，视角能自由旋转吗？**

回答：在2.5D俯视角模板中，摄像机视角是固定的，不支持自由旋转。但您可以调整其初始角度、距离和视野（FOV）  等参数。若项目需要自由旋转的摄像机，应考虑使用其他支持该功能的模板（如TPT）。

**9、问题：俯视角模板上下移动左右。走速度不等，上下移动角色不会跑起来怎么解决？**

回答：，核心原因是输入源对方向的速度处理不一致。解决方案是使用自制的UI左摇杆并正确配置输入映射。如果您在制作UI摇杆或配置脚本时遇到具体困难，建议参考官方模板中相关的UI控件和输入处理代码。

8.   → TaskManager 结束

### 示例2：区域停留时长任务

**场景** ：玩家在指定区域累计停留 N 分钟。

**Lua 代码** （放于 ugc/UGCGame/Task/SystemTask/BP_UGCAreaStayTimeTask.lua   ）：

local BP_UGCAreaStayTimeTask = {}

function BP_UGCAreaStayTimeTask:BeginPlay()
    print_dev("[BP_UGCAreaStayTimeTask:BeginPlay]")
    if not UE.IsValid(self.Owner) then return end
    if not self.Owner:HasAuthority() then return end

    self.PlayerEnterTime = {}

    -- 监听玩家进出区域（自定义消息）
    GMP.GlobalMessage.BindUObject(self, "UGC.CustomArea.PlayerEnter", self, self.OnPlayerEnterArea)
    GMP.GlobalMessage.BindUObject(self, "UGC.CustomArea.PlayerExit", self, self.OnPlayerExitArea)

    -- 每秒检查
    self.Timer = Timer.InsertTimer(1, function()
        local CurTime = UGCGameSystem.GetServerTimeSec()
        for PlayerKey, EnterTime in pairs(self.PlayerEnterTime) do
            if CurTime - EnterTime >= 60 then  -- 每60秒 +1
                self:AddStayTime(PlayerKey)
                self.PlayerEnterTime[PlayerKey] = CurTime
            end
        end
    end, true, "AreaStayTimer")
end

function BP_UGCAreaStayTimeTask:EndPlay()
    if self.Timer ~= nil then
        Timer.RemoveTimer(self.Timer)
        self.Timer = nil
    end
    if not UE.IsValid(self.Owner) then return end
    if not self.Owner:HasAuthority() then return end
    GMP.GlobalMessage.UnBindUObject(self, "UGC.CustomArea.PlayerEnter", self)
    GMP.GlobalMessage.UnBindUObject(self, "UGC.CustomArea.PlayerExit", self)
end

function BP_UGCAreaStayTimeTask:OnPlayerEnterArea(PlayerKey, AreaID)
    if self.AreaID and AreaID ~= self.AreaID then return end
    self.PlayerEnterTime[PlayerKey] = UGCGameSystem.GetServerTimeSec()
end

function BP_UGCAreaStayTimeTask:OnPlayerExitArea(PlayerKey, AreaID)
    self.PlayerEnterTime[PlayerKey] = nil
end

function BP_UGCAreaStayTimeTask:AddStayTime(PlayerKey)
    if not UE.IsValid(self.Owner) then return end
    local PC = UGCGameSystem.GetPlayerControllerByPlayerKey(PlayerKey)
    if not UE.IsValid(PC) then return end
    self.Owner:UpdateTaskProgressByID(self.TaskID, PC, 1, true)
end

function BP_UGCAreaStayTimeTask:ResetTaskForPlayer(PlayerKey)
    self.PlayerEnterTime[PlayerKey] = UGCGameSystem.GetServerTimeSec()
end

return BP_UGCAreaStayTimeTask

## 十四、常见问题

### Q: 为什么 BeginPlay 没执行？

检查蓝图是否正确绑定 Lua 脚本（文件名与蓝图名一致）
检查 GamePartConfig 中 TaskID 是否在任务线的 TaskIDList 中
检查 TaskType 中 TaskNode 是否正确指向蓝图实例

### Q: 为什么进度没更新？

确认self.Owner    和PC    都 Valid
确认在服务端执行（self.Owner:HasAuthority()   ）
确认消息路径正确，事件确实被触发
检查自定义属性的过滤条件是否匹配

## 5、行为树基础概念

**什么是行为树**

•用于控制AI决策逻辑的树状结构

•由根节点自上向下，自左向右执行

•将复杂AI行为拆分为原子节点

**四大节点类型**

•Composite 组合节点：控制执行流程

•Task 任务节点：执行具体动作

•Decorator 装饰器：条件判断与控制

•Service 服务节点：周期性数据更新

**核心组件关系**

•AIController：AI的大脑，运行行为树

•BehaviorTree：定义决策流程

•Blackboard：存储共享数据的记忆库

**设计优势**

•可视化编辑，直观表达AI逻辑

•模块化设计，节点可复用

•支持运行时调试与状态观察

## 6、黑板Blackboard详解

**黑板是什么**

•行为树的数据存储中心，类似记忆库

•以键值对(Key-Value)形式存储数据

•所有节点通过黑板键读写共享变量

**典型键值配置**

•Target：当前攻击目标（Actor）

•SelfActor：自身Actor引用（Actor）

•SpawnLoc：出生点坐标（Vector）

AttackInterval / AttackDistance：攻击参数（Float

**支持的数据类型**

•Boolean：是/否判断（如CanSeeEnemy）

•Enum：枚举状态（巡逻/追击/攻击）

•Vector / Object：位置和目标引用

•Float / Int：数值参数（距离/血量）

**数据流转机制**

•Service节点周期性写入检测数据

•Decorator读取黑板值做条件判断

•Task节点读取黑板获取目标执行动作

•实现节点间解耦的数据通信

## 7、Selector选择器节点

**工作原理**

•采用短路求值，从左向右依次评估子节点

•首个返回成功则立即停止并返回成功

•全部失败则整体返回失败

**典型应用场景**

•怪物AI优先级决策系统

•高优先级：发现敌人→发起攻击

•中优先级：听到声音→前去调查

•低优先级：无事发生→巡逻待命

**逻辑特点**

•类似编程中的 OR（或）逻辑

•优先级高的放左侧，低的放右侧

•最右边常放置保底方案（如Idle待机）

•任意子节点成功即跳过后续所有节点

**使用注意事项**

•子节点顺序直接影响行为优先级

•避免高优先级节点永远成功导致后续节点永远不执行

•可嵌套Sequence实现复合条件分支

## 8、Sequence序列节点

**工作原理**

•从左向右按顺序依次执行子节点

•任意一个子节点失败则立即停止，必须全部子节点都返回成功

•整体返回结果取决于最后执行状态

**典型应用场景**

•攻击流程：转向目标→移动到位→施放技能

•巡逻路径：移动到点A→等待→移动到点B

•条件检查：确认敌人存在→距离够近→执行攻击

•技能释放：选择技能→进入范围→播放动画→造成伤害

**逻辑特点**

•类似编程中的 AND（与）逻辑

•每一步都必须成功才能继续下一步

•适合表示有先后顺序的操作序列

**使用注意事项**

•子节点顺序代表操作执行的先后次序

•长序列中某步失败后需考虑恢复策略

•常与Selector组合使用实现完整决策

## 9、任务节点Task详解

**[Generic]导航移动到**

•利用导航网格(NavMesh)智能寻路

•自动避开障碍物选择最优路线

•支持配置停止距离和打断移动的状态

**[Generic]等待**

•等待时间（秒）：固定时间等待（秒）

•随机时间（秒）：随机偏差使行为自然

•应用：巡逻停留、攻击冷却、动画衔接

**[Generic]转向**

•控制怪物旋转面向目标方向

•参数支持 Vector位置 或 Actor 目标

•用于攻击前对准、交互前面对面等场景

**其他重要Task节点**

•Run Behavior Dynamic：运行子行为树（模块化复用）

•[Generic]寻找可达位置：查询指定范围中可以到达的位置

•[Generic]发送通知：向怪物脚本发送消息

## 10、装饰器节点Decorators

**装饰器概述**

•附着在Composite或Task节点上

•作为条件判断语句（类似IF）

•控制是否允许节点执行或中断执行

•是行为树灵活性的核心来源

**常用控制类装饰器**

•状态检查：检查当前怪物是否在某个状态中

•Generic_CheckDistance：检查怪物到某个点的距离

•[Generic]可见性检查：检查怪物是否可以看到某个点

**Blackboard条件装饰器**

•检查黑板变量值是否符合条件

•支持比较操作：等于/不等于/大于/小于

•最常用的装饰器类型

•例：Target不为空才执行攻击节点

**观察模式与中断**

•黑板值变化时可中断当前正在执行的任务

•实现即时响应（如敌人跑出范围立即停止追击）

•通知观察者参数决定中断时机

## 11、服务节点Services

**服务节点概述**

•附着在组合或者任务节点上运行

•以固定频率（Interval）周期性执行

•主要用来定期更新黑板数据

•不阻塞行为树的正常执行流程

**典型应用： [Generic]寻敌**

•定期调用仇恨感知系统检测周围敌人

•将最近的有效目标写入黑板Target键

•使行为树始终拥有最新的战场信息

**与Task节点的区别**

•Service是后台持续运行的辅助节点

•不影响父节点的执行结果

•在父节点活跃期间反复执行

**关键参数配置**

•Interval：执行间隔（秒），建议0.5~1.0

•RandomDeviation：增加随机偏差避免同步

•需注意频率过高会影响性能

**问题：怎么让怪物跳跃？**

回答：目前，编辑器中尚未直接开放“跳跃”这一独立的行为指令。虽然无法直接设置跳跃，但可以通过现有功能组合，模拟出类似的视觉效果，核心思路是利用 “传送”  功能来制造瞬间位移的“跳跃感”。在场景中规划好“起跳点”（A点）和“落地点”（B点）。B点可以设置在另一平台、高处或低处，以模拟不同跳跃轨迹。

function qqq:SetWaypoint(Waypoints, WayPointStrategy)
    local BlackBoard = AIBlueprintHelperLibrary.GetBlackboard(AIBlueprintHelperLibrary.GetAIController(self))
    BlackBoard:SetValueasBool("bUsePathPoint", true)
    local Tag = STExtraGameplayStatics.RequestGameplayTag("LogicPart.Class.FollowWaypoint", true)
    local LogicPart = self.LogicPartManagerComp:GetLogicPartByTag(Tag)
    if LogicPart then
        if WayPointStrategy == 0 then
            LogicPart:SetWaypoints(Waypoints,true,false)
        elseif WayPointStrategy == 1 then
            LogicPart:SetWaypoints(Waypoints,false,false)
        elseif WayPointStrategy == 2 then
            LogicPart:SetWaypoints(Waypoints,false,true)
        elseif WayPointStrategy == 3 then
            LogicPart:SetWaypoints(Waypoints,true,true)
        end
    end
end

## [近战技能经验贴](https://forum.developer.gp.qq.com/thread/1903)

**分类：** 技术答疑 | **浏览：** 60 | **发布：** 2026-03-20 10:50:50

## 技能背景

近战类型的技能是游戏中一类十分常见且通用的技能类型。一般来说，近战类型的技能需求描述通常都不复杂，基本上会以这样的范式进行叙述——玩家进行一个**XXXXXXXXXXX** 的动作，在**XXXXXXXXX** 时间段内，进行一个**XXXXXXXXXXXXX** 形状的伤害判定，最终对判定到的敌人造成若干伤害。

虽然通常技能的需求不难理解，但制作这样一个近战类型的技能，却有几类重要的问题需要解决。

### 精确的伤害判定

因为近战类型技能，往往会由一个具体的武器（刀、剑、枪、锤……）和对应该武器的攻击动作进行触发，而如果实际的伤害选取逻辑，不能很好的和该武器动作的运动轨迹匹配上的话，那么体验是十分糟糕的。

### 输入手感

一个近战的攻击技能通常将由多段（比如第一段横砍、第二段竖劈，第三段回旋斩）组成，为了玩家能连贯的使用这个技能，通常要考虑从输入到实际技能跳转的输入缓存、输入映射等问题，才能确保技能使用时手感的流畅。

### 命中手感

近战型的技能和一些枪械型技能不一样，由于玩家对于3D空间下距离、方向的判断很难准确，所以如果不做任何优化的话技能会很难命中，所以在实际制作近战技能的时候，需要辅导以位移吸附和朝向吸附来优化命中手感。

### 动作匹配

由于和平有不同Pose（站、蹲、趴），故需要考虑到近战技能可能在不同Pose下被使用，有可能需要在一个技能内实现多套动画和相应的伤害选取逻辑。

### 打击感

近战类型的技能会更加重视技能的表现和使用过程中的打击感。虽说总体而言，打击感是一个极其复杂的东西，如果要在这块凹极大的力气对于UGC玩法来说可能是得不偿失的——但通常来说，我们还是推荐开发者使用现在的这套框架，完成基础的打击感相关表现配置，这可以使得游戏的近战技能，得到一个具有一定下限的打击感。

## 实现思路

下面我们将依次来讲解要如何在绿洲编辑器里解决这些问题。

### 精确的伤害判定 - 实现解析

我们将使用专属的近战攻击节点来解决这个问题。

[图片#3893]

近战攻击节点的内部，提供了一套完整的近战选取——造成伤害的解决方案。首先是可以在节点上配置任意数量的攻击盒。

[图片#3894]

在游戏运行的时候，这些盒子会随着动画的播放不断运动，更新位置和朝向

[图片#3895]

而当这些盒子检测到目标时，会对目标造成配置好的伤害，并且在这个过程中，还具有去重的功能——即整个检测过程中，如果重复检测到一个目标，也只能造成一次效果。

### 输入手感 - 实现解析

为了达到流畅的手感，我们主要需要考虑连段跳转时的缓存问题。

#### 输入缓存

首先为什么需要有输入缓存？这是因为玩家的输入通常是不精确的，而连段的输入窗口有往往只有一段检测时机，如果玩家不能在那段时机正确按下输入，可能将错过连段输入，导致很强的挫败感。

[图片#3896]

为了解决这个问题，我们会利用到输入缓存节点，

[图片#3897]

输入缓存节点，在节点激活的时候会进入输入缓存，即这节点执行过程中的该输入都不会实际执行，而是被阻塞，直到节点结束时才再被执行。这样我们就能让一些在输入窗口之前的不精确输入实际生效了。

### 命中手感

在考虑一个近战类型的命中手感时，我们首先需要考虑这个会如何位移，然后再考虑基于这段位移如何校准命中，以及是否需要校准命中（吸附）。

#### 技能位移

一般我们会将技能的位移分为几种类型。

一、释放技能时允许同时使用左摇杆进行移动行为，一边移动一边攻击，这是由于移动的控制权还在玩家手里，所以一般选择不做任何吸附——因为吸附将抢夺玩家对移动的控制权，导致奇怪的手感。

二、释放技能时不允许玩家同时使用左摇杆进行移动，只能原地使用某个攻击动作。这时候可以选择做吸附，且一般会选择在实际技能效果生效之前的一段时间窗口做吸附，且一般的，这种通常就是由代码对角色的位置和朝向进行修正，我们一般称为**程序吸附** 。

三、释放技能时不允许玩家同时使用左摇杆进行移动，但同时会由动作本身承担一段动画里的固定位移，一般我们也称为RootMotion。这时候也可以选择做吸附，但原理和上面的那种方式不太相同，本质相当于根据希望到达的目标点，修正原本的固定位移——因为只有保证根据原本的位移来进行等比例的修正，才能确保整体的位移节奏可以和动作对的上，这种我们一般称为**美术吸附** 。

#### 技能吸附

程序吸附的使用

### 打击感

#### 打击感简易公式

我们在UGC玩法里不必过分追求打击感，否则实现成本会十分高昂，但是依旧有一些简单、容易做到的方式提升游戏表现，具体来说可以总结为：

**顿帧** + **受击效果（震屏、音效、特效、闪白、击退）**

顿帧：命中目标时让自身/受击者的动画速率临时慢下来，持续一个较短的时间。这将让近战的命中看上去有一种卡肉感，即运动好像真的被命中目标所阻塞。

[图片#3907]

配置近战攻击节点里的顿帧相关参数，即可实现顿帧效果，顿帧的持续时间不宜过长，这是因为顿帧实际上只是一个客户端表现，ds并不受顿帧影响，技能是会继续往下走的，如果顿帧持续时间过长，将导致后续技能表现可能不匹配。

受击效果：即命中后让受击者出现的一系列表现向相关效果，可有效提升某次命中的具体打击表现。

[图片#3908]

在近战节点中启用受击效果，然后配置相应的受击资产，即可让对应的近战命中敌人时，产生一系列对应的受击效果。

[图片#3909]

震屏：可以配置CameraShake相关达到受击震屏效果，命中后，将对受击者生效一次对应的屏幕抖动。

音效：可以配置Sound相关来达到音效相关效果。命中后，将播放一个对应的音效。如果受击信息里有受击部位的区分，则会根据配置的音效区分播爆头音效或是普通音效。

特效：可以配置Particle相关来达到相关效果。命中后，将在受击者的受击部位播放一个对应的受击特效。若受击信息里无部位信息，则只会在受击原地播放该特效。

闪白：可以配置Fresnel相关来达到相关效果。命中后，将使受击者的模型材质出现边缘高亮光，从而表现出受击的命中效果。

击退：可以配置HitBack来达到相关效果。使得受击者受到一段位移上的击退效果。

## 如何获取武器网格组件

粒子特效需要"挂"在武器的 3D 模型上。武器的 3D 模型就是**网格组件（MeshComponent）** 。我们必须先拿到它，才能把特效附着上去。

self.mesh = self:GetWeaponMeshComponent()

 self   指的是当前武器 Actor 自身
 GetWeaponMeshComponent()   是武器基类提供的方法，返回武器的静态网格组件。
 GetWeaponSkeletalMeshComponent()   是武器基类提供的方法，返回武器的骨骼网格组件。

武器基类提供了 OnWeaponMeshLoadFinished   回调函数，当武器网格模型加载完成时会自动触发。在这个回调中获取网格组件是**最可靠** 的方式，无需手动轮询重试。

-- 武器网格加载完成的回调（引擎自动调用）
function W_Melee:OnWeaponMeshLoadFinished(SlotID, IsEquipped)
    local meshComp = self:GetWeaponMeshComponent()
    if meshComp then
        self.mesh = meshComp
        self:OnMeshReady()  -- 网格就绪，执行后续逻辑
    end
end

💡 **说明** ：OnWeaponMeshLoadFinished   的参数：

 SlotID  ：武器槽位 ID
 IsEquipped  ：是否已装备

local red   = { R = 1, G = 0, B = 0, A = 1 }  -- 红色

## 设置颜色循环渐变

时间轴：|--- 红→绿 ---|--- 绿→蓝 ---|--- 蓝→红 ---|--- 红→绿 ---|...
        0s          2s          4s          6s          8s
        ||

把一个循环分成 3 个阶段，每个阶段在两种颜色之间做**线性插值（Lerp）**
用定时器每 0.05 秒更新一次颜色（20fps 的更新频率）
通过SetColorParameter   把计算出的颜色设置到粒子上

### 配置参数

在脚本顶部定义可调参数：

local W_Melee = {
    COLOR_CYCLE_DURATION = 6.0,   -- 一个完整循环的时长（秒），越大越慢
    COLOR_UPDATE_INTERVAL = 0.05, -- 颜色更新间隔（秒），越小越平滑
}

### 颜色插值函数

-- 在两个颜色之间做线性插值
-- colorA: 起始颜色, colorB: 目标颜色, t: 进度 (0.0 ~ 1.0)
function W_Melee:LerpColor(colorA, colorB, t)
    return {
        R = colorA.R + (colorB.R - colorA.R) * t,
        G = colorA.G + (colorB.G - colorA.G) * t,
        B = colorA.B + (colorB.B - colorA.B) * t,
        A = colorA.A + (colorB.A - colorA.A) * t,
    }
end

💡 **线性插值（Lerp）** 就是：  结果 = 起点 + (终点 - 起点) × 进度  。当 t=0 时结果等于起点，t=1 时结果等于终点，t=0.5 时结果是两者的中间值。

### 启动颜色循环

function W_Melee:StartColorCycle()
    -- 先停止已有的循环（防止重复启动）
    self:StopColorCycle()

    -- 记录开始时间
    self.colorCycleStartTime = os.clock()

    -- 定义关键帧颜色
    local keyColors = {
        { R = 1, G = 0, B = 0, A = 1 }, -- 红色
        { R = 0, G = 1, B = 0, A = 1 }, -- 绿色
        { R = 0, G = 0, B = 1, A = 1 }, -- 蓝色
    }

    -- 创建定时器，每隔 0.05 秒更新一次颜色
    self.colorCycleTimer = UGCTimerUtility.CreateLuaTimer(
        self.COLOR_UPDATE_INTERVAL,
        function()
            -- 安全检查：粒子组件是否还有效
            if not self.fireParticleComp then
                self:StopColorCycle()
                return
            end

            -- 计算当前进度（0.0 ~ 1.0 循环）
            local elapsed = os.clock() - self.colorCycleStartTime
            local progress = (elapsed % self.COLOR_CYCLE_DURATION) / self.COLOR_CYCLE_DURATION

            -- 确定当前在哪个阶段（0=红→绿, 1=绿→蓝, 2=蓝→红）
            local phase = progress * 3
            local phaseIndex = math.floor(phase)
            local phaseT = phase - phaseIndex  -- 当前阶段内的进度

            if phaseIndex >= 3 then
                phaseIndex = 2
                phaseT = 1.0
            end

            -- 计算当前颜色
            local fromColor = keyColors[phaseIndex + 1]
            local toColor = keyColors[((phaseIndex + 1) % 3) + 1]
            local currentColor = self:LerpColor(fromColor, toColor, phaseT)

            -- 设置到粒子上
            self.fireParticleComp:SetColorParameter("Color", currentColor)
        end,
        true,                    -- 循环执行
        "W_Melee_ColorCycle"     -- 定时器名称
    )
end

### 停止颜色循环

function W_Melee:StopColorCycle()
    if self.colorCycleTimer then
        UGCTimerUtility.RemoveLuaTimer(self.colorCycleTimer)
        self.colorCycleTimer = nil
    end
end

### 清理工作（非常重要！）

武器被销毁时，**必须** 清理所有定时器和粒子组件，否则会导致内存泄漏或空引用报错：

function W_Melee:ReceiveEndPlay()
    -- 1. 停止颜色循环定时器
    self:StopColorCycle()

    -- 2. 销毁粒子组件
    if self.fireParticleComp then
        self.fireParticleComp:Deactivate()
        self.fireParticleComp = nil
    end

    -- 3. 调用父类清理
    W_Melee.SuperClass.ReceiveEndPlay(self)
end

## [【策划面对面01】编辑器高频需求研发规划](https://forum.developer.gp.qq.com/thread/1891)

**分类：** 创意交流 | **浏览：** 130 | **发布：** 2026-03-17 15:31:26

各位开发者，大家好。

随着CG36版本编辑器即将上线，我们收到了大量关于新功能与开发体验的需求反馈。本期策划面对面，我们也会和大家同步开发者们之前通过问卷提到的部分问题的未来规划。好让大家能提前掌握后续更新节点，以及可制作的功能与玩法。

#### 一、 美术资源：载具资源

**1、高频需求：做古代国风类项目急需“马”的载具模板，何时能提供？**

回答：**CG38** 版本左右会正式支持。我们在“资源管理器”的模板库中更新了基础载具组件，其中包括可用于搭建坐骑系统的“马”的骨骼模型与基础操控逻辑开发者们可以直接引用并依据项目需求，调整外观、速度与操控反馈参数。详细的使用文档可在CG38左右更新后的开发者Wiki中查看。

**2、 高频需求：增加飞行类载具模板。**

回答：飞行类载具已在规划中，预期**CG37** 版本（6月份版本）会更新，具体进度我们将适时同步。

**3、高频需求：增加更多玩家动作和技能特效资源。**

回答：我们会坚持在每个版本，配套推出新的技能模板及相关的动作、特效资源，丰富大家的创作素材库并持续更新中。

#### 二、工具链：AI助手

**1、高频需求：AI开发助手，知识库老旧，错误率高，有何改进计划？**

回答：目前，新一代AI助手的开发方案已在排期中。新方案将着重于提升代码建议的准确性与上下文理解能力，旨在为大家提供一个更可靠的智能编程伙伴。具体进度我们将适时同步。

#### 三、需求与解答：功能与模版

**1、高频需求：是否允许对更多官方资产进行编辑？**

回答：出于保障官方资源完整性及避免兼容性问题的考虑，核心官方资产目前仍保持只读模式。但我们持续通过发布可直接使用并修改的“可编辑模板”（如CG38的载具模板）来满足大家的定制化需求。我们会评估在两者间取得最佳平衡的方案。

**2、高频需求：表格编辑器：功能孱弱，无法批量修改、删除，建议支持导出为Excel/JSON进行批量管理。**

回答：当前，表格编辑器已支持将数据导出为Excel格式，方便大家进行本地批量操作与存档。关于编辑器内更强大的批量操作功能，我们已纳入需求池进行评估，欢迎大家持续反馈。

**3、高频需求：尽可能让更多配置（如背包道具）支持表格化导入导出，极大提升配置效率。**

回答：DataTable (DT) 已支持导出为Excel，这正是为了实现高效的批量配置。我们后续会推动更多系统配置支持此标准化流程。

**4、高频需求：官方模板常引用旧版或缺失资源，导致无法直接使用。**

回答：对于因历史原因导致的模板资源缺失或引用错误问题，我们表示歉意。我们已启动对老旧模板的系统性清理与下线工作。同时，我们建立了更严格的模板质检流程，确保新发布模板的可用性。如果您遇到具体模板问题，欢迎反馈具体名称，我们将优先处理。

**5、高频需求：调试依赖打印Log，效率低下，有无可视化调试器（Profiler）计划？**

回答：当前CG36版本提供了纯净版日志展示功能，开发者通过调用接口打印日志时，会实时显示在编辑面板上，且过滤了底层C++日志，只输出开发者自定义日志和引擎Lua日志。后续会支持AI解读日志等功能。

**后续规划：**

自定义打印级别：该功能已列入规划，后续版本将推进。

全量调试功能：计划在后续版本加入服务端调试，实现全量调试支持。

#### 四、 未来规划展望

我们始终关注着大家对于更丰富玩法、更强大工具与更紧密社交的期待。以下是我们基于开发者们问卷反馈，正在规划与探讨中的方向：

**1、载具系统的扩展**

● 大家的期待：除陆地载具之外，增加飞行类载具模板，为玄幻、科幻等题材打开更多想象空间。

● 官方规划：在CG37夯实基础载具能力后，扩展更多载具类型已在我们的研发路线图上。我们将评估具体技术方案与资源需求，在合适的版本中为大家带来更新颖的移动与战斗体验。

**2、商业化设计的深化**

● 大家的期待：在现有商城模板基础上，提供更多高转化率的商业化系统模板，如首充奖励、随机宝箱等，帮助团队快速搭建成熟的商业循环。

● 官方规划：我们完全认同，强大的商业化工具是支撑优秀作品持续运营的关键。团队正在系统性地梳理与设计一系列“可复用的商业化模块”。我们的目标是让开发者不仅能便捷调用功能，更能理解其背后的设计逻辑与数据平衡，其他商业化模板将在后续版本中陆续推出。

**3、社交功能的强化**

● 大家的期待：希望有游戏内便捷邀请好友的功能，降低组队门槛，增强游戏的社交传播与粘性。

● 官方规划：强化玩家间的实时互动是我们长期投入的方向。游戏内的一键邀请、房间快速加入等社交功能，已在技术可行性研究与原型设计阶段。我们将结合平台能力，致力于为各位的作品注入更便捷、有趣的社交动力。局内聊天，添加好友等功能预期37版本-38版本上线。

**4、 UI美术与素材效果**

● 大家的期待：希望能更灵活地自定义或隐藏游戏内基础UI组件（如准星、方位标尺），以便为剧情、影视化运镜或特殊玩法提供更纯净的屏幕表现，实现更高的美术自由度。

● 官方规划：我们非常支持开发者追求个性化的视觉表达。为此，我们已在规划提供相应的API，以支持在游戏中动态控制准星等核心UI元素的显示与隐藏。我们也在持续迭代UI相关功能，大家如有发现未能隐藏或调整的和平UI，可反馈致官方，将进行处理。

**5、 官方论坛的易用性**

● 大家的期待：更方便自行查询问题，更快速高效的学习与解决问题。

● 官方规划：社区论坛也在进一步规划改善中，后续将引用全新的ai模型，以及重新调整全新的入门教学WIKI与视频。

您的反馈至关重要

以上是我们基于当前大家最集中的声音所制定的部分规划。编辑器优化，始终离不开每一位开发者。如果您还有其他具体的功能需求或优化建议，欢迎在本帖下方回复告诉我们。 我们将持续收集，并作为未来优化的重要参考。感谢您的认真聆听与答复！

如大家有其他对编辑器的需求，可以在本帖里回复，我们会持续收集大家的建议。

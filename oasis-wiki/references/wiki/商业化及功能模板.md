# 商业化系统

绿洲启元商业化系统是提供给玩法的付费功能解决方案，开发者可以根据实际的玩法需求，定制适合玩法自身的付费商品（商品形式不限），而玩家则能够使用绿洲官方货币购买相关付费商品，且商品购买后将保存在玩家账户（对局结束也将保留）。

> **商业化权限可联系官方人员进行咨询与申请开通**

<br>

## 系统概述

### 货币及商品购买

绿洲官方货币包括：绿洲币和启元币。
- 绿洲币： 玩家充值获得
- 启元币： 玩家完成绿洲任务获得

玩法接入绿洲启元商业化功能后，玩家可以通过两种方式进行付费商品的购买：

**1. 玩法详情页购买**

无需进入玩法就能够直接购买商品的途径，可支持绿洲币、启元币购买付费商品。

![image.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/q2PGKimage.png)

**2. 玩法内购买**

开发者在玩法内可以自行设计商城的购买方式，仅支持绿洲币购买付费商品。

---

### 系统交互流程

商业化系统给玩法提供了获取和修改玩家账户内自定义商品数据的能力：
- 商品购买接口和物品使用接口：玩法可以向玩家账户添加指定的商品和扣除指定物品，添加商品会根据商品的价格信息扣除玩家对应数额的绿洲币
- 数据获取接口：玩法可以获取到玩家账户内于该玩法定义的商品数据和物品数据，玩法可以使用这些数据展示和制作相关的功能

由于对玩家账户的数据读取和修改都需要在游戏后台服务器进行，所以玩法需要先注册购买/使用商品的协议，调用绿洲启元商业化系统的接口时，会根据协议指定的购买/使用的商品ID和数量，向后台服务器发起购买/使用请求：

![image.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/6xaVWimage.png)

更多商业化相关的API可以查看 [**UGCCommoditySystem类**](https://developer.gp.qq.com/api/#/searchContent/UGCCommoditySystem?classDetailShow=true&path=class%2Fdetail%2F%E5%92%8C%E5%B9%B3%E5%85%A8%E5%B1%80%E6%8E%A5%E5%8F%A3%2F%E5%95%86%E4%B8%9A%E5%8C%96%E4%B8%8E%E5%8A%9F%E8%83%BD%E6%A8%A1%E6%9D%BF%2FUGCCommoditySystem.json&isSelect=1&apiEnc=%5B%22%E7%B1%BB%22%5D&apiLabel=UGCCommoditySystem)

## 系统功能接入

### 商业化数据配置

商品数据依赖于编辑器内的两张配置表：<span style="color:red;">``物品表（UGCObject）``</span> 与 <span style="color:red;">``商品表（UGCShop）``</span>

> 注意区分物品和商品概念：直接售卖的是商品，购买商品后获得的是物品

玩法工程默认情况下不存在以上配置表，需要通过编辑器菜单栏下的 ``窗口 -> Create System Default Table -> Create Product Table`` 手动创建，创建后在工程目录下会自动生成Data目录，两张配置表在``Asset/Data/Table``目录中：

![image.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/ZwX1Iimage.png)
![image.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/fzMQnimage.png)

#### 物品表（UGCObject）

用于配置玩法内的物品信息，在该表格配置后才能在商品表中配置商品信息，否则将会被校验为非法物品导致购买失败

![image.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/FkcZXimage.png)

|配置|字段名|字段类型|说明|备注|
|-|-|-|-|-|
|小icon|ItemSmallIcon|string|物品图标|物品icon图引用路径（FSoftObjectPath）|
|项目ItemID|ItemID|number|物品ID|限制在1001-10000|
|物品名称|ItemName|string|物品名称信息|限制15个字符|
|物品描述|ItemDesc|string|物品描述信息|限制60个字符|

#### 商品表（UGCShop）

用于配置玩法内正式售卖的商品，商品在该表格配置后才能售卖，配置错误或未按规则配置(如配置的物品ID不存在)都将导致购买失败

|配置项|字段名|字段类型|说明|备注|
|-|-|-|-|-|
|商品ID|ProductId|number|商品ID|限制在9000000-9999999|
|商品类型	|ProductType|number|该商品的类型	|物品/货币|

|售卖价格|SellingPrice|number|售卖价格|
|限购类型|LimitType|number|限购类型|不限购/每日限购/每周限购/永久限购|
|限购数量	|PurchaseLimit|number|限购数量|
|上架时间	|ListingTimeString|string|商品上架时间|限时售卖的商品上架时间|
|下架时间	|DelistingTimeString|string|商品下架时间|限时售卖的商品下架时间|
|商城ID	|StoreId|number|商品售卖的方式	|战斗内/玩法详情页|
|商品排序|SortPriority|number|商品在详情页/商城的排序序号|数值越小，排序越靠前|
|商品包含物品数量|ItemNum|number|一件商品中包含的物品数量||
|消耗物品ID|CostID|number|自定义货币关联的物品ID||
|折扣|Discount|number|折扣点数|0代表不打折|
|折扣开始时间|DiscountStartTime|string|折扣生效的开始时间|只需填写年、月、日、时|
|折扣结束时间|DiscountEndTime|string|折扣结束的时间|只需填写年、月、日、时|
|商城页签ID|TabID|number|``Shop_MainUI_UIBP`` 中配置的TabID|用于商品的分组|

### 功能逻辑接入

商业化的功能逻辑主要包含：
- 调用 ``商品购买接口`` 和 ``物品使用接口``分别进行商品的购买获取和物品的使用消耗
- 绑定 ``商品购买结果委托`` 和 ``物品使用结果委托`` 分别处理商品的购买结果逻辑和物品使用结果逻辑

<br>

~~注册商品购买协议和物品使用协议~~（已废弃）
> <font color='#A52A2A'>注册过购买和使用协议的玩法请保留原接入方式，无需更改</font>

<font color='#DDDDDD'>
商品购买和物品使用行为都是在客户端进行处理，而购买和使用通过发送协议来实现，所以需要先在DS内注册协议后，再从客户端调用接口来触发购买商品和使用物品。
购买/使用协议包含商品/物品的ID以及单次购买/使用的数量，根据玩法需要，设计玩法中可能出现的商品类型与单次购买/使用数量组合。

**商品购买协议**


---注册商品购买协议
---生效范围：S
---@param BuyCommodityCMD string  @使用协议
---@param ProductID int  @商品的ID
---@param Count int  @单次购买的数量
function UGCCommoditySystem.RegBuyCMD(BuyCommodityCMD, ProductID, Count)

- BuyCommodityCMD： 待注册的购买协议，调用购买商品接口时，会根据传入的协议发送商品购买请求
- ProductID： 商品的ID，在商品表（UGCShop）中配置
- Count： 协议发送时，单次购买商品的数量


**物品使用协议**

---注册使用物品协议
---生效范围：S
---@param UseCommodityCMD string  @使用协议
---@param CommodityID int  @物品的ID
---@param Count int  @单次消耗的数量
function UGCCommoditySystem.RegUseCMD(UseCommodityCMD, CommodityID, Count)

- UseCommodityCMD： 待注册的使用协议，调用使用物品接口时，会根据传入的协议发送物品使用请求
- CommodityID： 物品的ID，在物品表（UGCObject）中配置
- Count： 协议发送时，单次使用物品的数量

> **注意：**
商品购买协议注册接口传入的ProductID是商品表中的商品ID，而物品使用协议注册接口传入的CommodityID是物品表的物品ID，注意两者参数的区别
</font>

#### 1. 购买商品
开发者可以根据玩法需求自定义商品信息在游戏内的展示形式，例如制作自己的商店界面来展示商品的图标和价格等信息；商品信息内容需要与物品表、商品表的数据配置保持一致。（若不一致，发生实际行为后将以配置为准）

在需要执行商品购买处调用接口 [``UGCCommoditySystem.BuyUGCCommodity2``](https://developer.gp.qq.com/api/#/searchContent/UGCCommoditySystem?classDetailShow=true&path=class%2Fdetail%2F%E5%92%8C%E5%B9%B3%E5%85%A8%E5%B1%80%E6%8E%A5%E5%8F%A3%2F%E5%95%86%E4%B8%9A%E5%8C%96%E4%B8%8E%E5%8A%9F%E8%83%BD%E6%A8%A1%E6%9D%BF%2FUGCCommoditySystem.json&isSelect=1&apiEnc=%5B%22%E7%B1%BB%22%5D&apiLabel=UGCCommoditySystem&autoJump=BuyUGCCommodity2)，例如玩家点击了玩法内的购买按钮时，调用接口后系统会弹出购买的二次确认界面，玩家点击“购买”按钮后，客户端会发送购买协议至后台进行购买处理。

商品购买二次确认弹窗界面：

![image.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/bfY2dimage.png)

如果购买流程正常且商品信息校验正确，会提示购买道具成功；而若购买失败，系统会根据失败类型弹出购买失败的提示，例如绿洲币不足。

> 购买成功提示

![image.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/2Xg4Simage.png)

> 购买失败提示

![image.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/Zwmilimage.png)

若开发者需要对购买的结果进行额外的逻辑处理，例如根据购买结果更新玩家拥有的商品信息显示，则可以绑定购买结果的委托 [``UGCCommoditySystem.BuyUGCCommodityResultDelegate``](https://developer.gp.qq.com/api/#/searchContent/UGCCommoditySystem?classDetailShow=true&path=class%2Fdetail%2F%E5%92%8C%E5%B9%B3%E5%85%A8%E5%B1%80%E6%8E%A5%E5%8F%A3%2F%E5%95%86%E4%B8%9A%E5%8C%96%E4%B8%8E%E5%8A%9F%E8%83%BD%E6%A8%A1%E6%9D%BF%2FUGCCommoditySystem.json&isSelect=1&apiEnc=%5B%22%E7%B1%BB%22%5D&apiLabel=UGCCommoditySystem)

```lua
--绑定委托
UGCCommoditySystem.BuyUGCCommodityResultDelegate:Add(OnBuyUGCCommodityResult, Tester)

--委托解绑
UGCCommoditySystem.BuyUGCCommodityResultDelegate:Remove(OnBuyUGCCommodityResult, Tester)
```

#### 2. 使用物品
玩家所拥有的商品信息，在游戏内开发者可以根据玩法需求自定义显示形式，比如展示在特定的UI面板内。

在需要执行物品使用处调用接口 [``UGCCommoditySystem.UseUGCCommodity2``](https://developer.gp.qq.com/api/#/searchContent/UGCCommoditySystem?classDetailShow=true&path=class%2Fdetail%2F%E5%92%8C%E5%B9%B3%E5%85%A8%E5%B1%80%E6%8E%A5%E5%8F%A3%2F%E5%95%86%E4%B8%9A%E5%8C%96%E4%B8%8E%E5%8A%9F%E8%83%BD%E6%A8%A1%E6%9D%BF%2FUGCCommoditySystem.json&isSelect=1&apiEnc=%5B%22%E7%B1%BB%22%5D&apiLabel=UGCCommoditySystem)，例如玩家点击了玩法内的使用按钮时，调用接口时可以决定是否弹出使用物品的二次确认界面：若不弹，则直接发送协议至后台，进行物品校验的结算；若弹出二次确认界面，则玩家点击使用按钮后才会发送协议。

使用物品前的二次确认弹窗界面：

![image.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/Jr3nWimage.png)

当物品数量足够且校验通过时，会提示“使用道具成功”；若使用失败，系统会根据失败类型弹出使用失败的提示，例如“道具数量不足”。

> 使用成功提示

![image.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/WA5sLimage.png)
	
> 使用失败提示

![image.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/5FMPximage.png)

若开发者需要对使用的结果进行逻辑处理，例如物品扣除成功时触发物品的使用效果，则可以绑定使用结果的委托 [``UGCCommoditySystem.UseUGCCommodityResultDelegate``](https://developer.gp.qq.com/api/#/searchContent/UGCCommoditySystem?classDetailShow=true&path=class%2Fdetail%2F%E5%92%8C%E5%B9%B3%E5%85%A8%E5%B1%80%E6%8E%A5%E5%8F%A3%2F%E5%95%86%E4%B8%9A%E5%8C%96%E4%B8%8E%E5%8A%9F%E8%83%BD%E6%A8%A1%E6%9D%BF%2FUGCCommoditySystem.json&isSelect=1&apiEnc=%5B%22%E7%B1%BB%22%5D&apiLabel=UGCCommoditySystem)

```lua
--绑定委托
UGCCommoditySystem.UseUGCCommodityResultDelegate:Add(OnUseUGCCommodityResult, Tester)

--委托解绑
UGCCommoditySystem.UseUGCCommodityResultDelegate:Remove(OnUseUGCCommodityResult, Tester)
```

#### 3. 获取商品数据与物品数据

在实际的商业化玩法中，可能需要获取当前玩家的商品信息和物品信息用于玩法自定义的UI展示，可以通过提供的商品信息获取接口和物品信息获取接口进行数据的获取。

**物品数据获取**

- 获取所有玩家的物品数据
[``UGCCommoditySystem.GetAllPlayerUGCCommodityList``](https://developer.gp.qq.com/api/#/searchContent/UGCCommoditySystem?classDetailShow=true&path=class%2Fdetail%2F%E5%92%8C%E5%B9%B3%E5%85%A8%E5%B1%80%E6%8E%A5%E5%8F%A3%2F%E5%95%86%E4%B8%9A%E5%8C%96%E4%B8%8E%E5%8A%9F%E8%83%BD%E6%A8%A1%E6%9D%BF%2FUGCCommoditySystem.json&isSelect=1&apiEnc=%5B%22%E7%B1%BB%22%5D&apiLabel=UGCCommoditySystem&autoJump=GetAllPlayerUGCCommodityList)

	> 当所有玩家的 ``UGCPlayerController:Server_OnUGCCommodityPlayerDataReady()`` 触发后，才可以使用该函数获取所有用户数据，请务必注意调用时序
	

### 绿洲币的显示与充值

**1. 显示并获取绿洲币余额及充值入口**

- 绿洲币余额和充值界面默认不显示，需要显式调用接口 [``UGCCommoditySystem.ShowRechargeEntryUI``](https://developer.gp.qq.com/api/#/searchContent/UGCCommoditySystem?classDetailShow=true&path=class%2Fdetail%2F%E5%92%8C%E5%B9%B3%E5%85%A8%E5%B1%80%E6%8E%A5%E5%8F%A3%2F%E5%95%86%E4%B8%9A%E5%8C%96%E4%B8%8E%E5%8A%9F%E8%83%BD%E6%A8%A1%E6%9D%BF%2FUGCCommoditySystem.json&isSelect=1&apiEnc=%5B%22%E7%B1%BB%22%5D&apiLabel=UGCCommoditySystem&autoJump=ShowRechargeEntryUI)，接口将会返回充值入口UI的对象实例

- 充值入口UI将显示玩家绿洲币余额，点击入口的 ``+`` 号，将弹出和平的绿洲币充值界面<br>
![image.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/hduoAimage.png)

**2. 调整充值入口UI（可选）**

开发者可以保存调用接口获取到的UI对象实例，若需要隐藏入口或者调整入口位置，可以调用UI相关的接口对入口UI的对象实例进行操作。

```lua
-- 显示并获取界面对象实例
UGCCommoditySystem.ShowRechargeEntryUI():Then(
	function (Result)
		local UI = Result:Get()
		UI:SetVisibility
	end
)
```

**注意事项**

1. PIE调试和外研线测试期间，充值入口的 ``+`` 号不会显示，也无法弹出绿洲币充值界面
2. 发布上传且开通内购权限的工程会显示绿洲币余额，否则不予显示


绑定购买商品结果委托与回调逻辑处理

```lua
	-- 购买商品后的结果，通过UGCCommoditySystem.BuyUGCCommodityResultDelegate返回，可以在GameState中增加和移除委托监听
	function UGCGameState:ReceiveBeginPlay()
			UGCGameState.SuperClass.ReceiveBeginPlay(self)
			UGCCommoditySystem.BuyUGCCommodityResultDelegate:Add(self.OnBuyUGCCommodityResult, self)
	end

	function UGCGameState:ReceiveEndPlay()
			UGCGameState.SuperClass.ReceiveEndPlay(self)
			UGCCommoditySystem.BuyUGCCommodityResultDelegate:Remove(self.OnBuyUGCCommodityResult, self)
	end

	--可以通过参数得知玩家物品的变化已经是否购买成功
	function UGCGameState:OnBuyUGCCommodityResult(bSucceeded, PlayerKey, CommodityID, Count, UID, ProductID)
			local Result = string.format(
					"(UGCGameState:OnBuyUGCCommodityResult bSucceeded=%s PlayerKey=%s CommodityID=%s Count=%s)", 
					tostring(bSucceeded), 
					tostring(PlayerKey), 
					tostring(CommodityID), 
					tostring(Count),
					tostring(ProductID)
			)
			ugcprint(Result)

			local cond = UGCGameSystem.IsServer()
			if cond then
					local Result = UGCCommoditySystem.GetAllPlayerUGCCommodityList()
					local PlayerController = UGCGameSystem.GetPlayerControllerByPlayerKey(PlayerKey)
					ugcprint(ToTreeString(Result[PlayerController:GetInt64UID()]))
			else
					local Result = UGCCommoditySystem.GetUGCCommodityList()
					ugcprint(ToTreeString(Result))
			end
	end
```

### 编辑器PIE调试
- 编辑器环境下可以在PIE时模拟购买商品、使用商品的功能，按正常流程接入即可在PIE时正常测试

## 模板结构

模板结构划分为五个部分：商城标题、货币类型、商品页签、商品橱窗及商品详情。

![image.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/nFYj5image.png)

**A. 商城标题**

配置商城名称及商城的icon图
- 名称限制：7个字以内
- icon规格：31*31（支持jpg、png格式），2MB以内

**B. 货币类型**

商品支持绿洲币购买，开发者也可以自定义专属货币，且货币属于物品类别，因此货币的配置复用 [物品表](https://developer.gp.qq.com/wikieditor/#/catalog/20008?autoJump=%E7%89%A9%E5%93%81%E8%A1%A8%EF%BC%88UGCObject%EF%BC%89)
> 自定义货币定位为二级货币，获取渠道需要设计为通过绿洲币购买获得

**C. 商品页签**

开发者可以对售卖商品进行分类，每种类别对应一个页签，例如：“武器”、“消耗品”等
- 页签字数限制：4个字以内
> 至少需要配置一种页签

**D. 商品橱窗**

橱窗是向玩家展示可售卖商品的区域，每件商品的展示内容包含：商品基础信息、限购信息、限时售卖信息、折扣信息及售卖状态。

![image.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/7MmEjimage.png)

【商品基础信息】

商品本质上是售卖的物品，因此每件商品的基础信息实际就是物品的信息，物品ID、物品名、物品描述以及物品icon图等均配置在 [物品表](https://developer.gp.qq.com/wikieditor/#/catalog/20008?autoJump=%E7%89%A9%E5%93%81%E8%A1%A8%EF%BC%88UGCObject%EF%BC%89) 中

【限购信息】

支持限购类型及限购数量的配置：
- 每日限购：每天限购固定数量，0点刷新
- 每周限购：每周限购固定数量，每周一0点刷新
- 永久限购：玩法内限购固定数量

【限时售卖信息】

# 抽奖模板

抽奖是商业化玩法中常见的营收形式之一，通常提供包括各类普通奖励、稀有奖励和保底奖励等来吸引玩家参与。形式、规则、机制和奖品共同构成了抽奖功能的要素，抽奖模板提供了完善的解决方案，开发者可以通过配置化的方式快速搭建抽奖系统，也能定制个性化的抽奖需求。

> 抽奖模板已经历过重构优化，不再向后兼容，开发者如果曾经下载过旧版本，需从 [内容浏览器](https://developer.gp.qq.com/wikieditor/?timeStamp=1725589224096#/catalog/124?autoJump=%E5%86%85%E5%AE%B9%E6%B5%8F%E8%A7%88%E5%99%A8) 右键删除 ``ExtendResource`` 目录下对应的模板文件夹，再下载最新版本并按照本文档配置使用

<br>

## 功能概要

完整的抽奖功能包含抽奖奖池、抽奖货币、掉落模组等部分，买赠礼包和兑换物商城为额外可选配置功能，抽奖涉及的各类奖品物资均属于物品。

![image.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/EKWaaimage.png)

**抽奖货币**

玩家可以通过自定义的二级货币进行抽奖，自定义货币本质上也是一类物品，与普通商城物品一样能通过绿洲币进行购买转换，也支持做首抽折扣。

**抽奖奖池**

开发者可以配置多个奖池，每个奖池通过不同的抽奖页签进行切换展示。玩家使用抽奖货币进行抽奖，由掉落模组决定每次抽奖的产出物结果，奖品可以是游戏内的道具物品，也可以是兑换物。

**买赠礼包**

开发者可以配置累抽的礼包赠品，以提升玩家的抽奖体验。

**兑换物商城**

抽奖奖池产出的兑换物，可用于兑换商城进行商品的兑换，作为对抽奖产出物的消耗渠道的补充。

<br>

## 快速开始

### 1. 下载抽奖模板

抽奖模板通过 [资源商店](https://developer.gp.qq.com/wikieditor/#/catalog/336) 的形式提供下载，可以在资源商店中搜索“抽奖模板”找到该模板，点击“下载资源”把模板下载至本地工程。

![image.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/44Ooximage.png)

下载后的模板将以 ```ExtendResource``` 资源包的形式追加到工程里。

![image.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/l6oHnimage.png)

---

### 2. 创建奖品

在抽奖模板中，无论是奖品还是抽奖所用的货币，亦或是商城兑换奖品的碎片，本质上都是一个物品，在正式配置抽奖模板之前，需要先将这些作为物品配置完全。

点击编辑器菜单栏 ``窗口 -> Create System Default Table -> Create Product Table``。

![image.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/ZiKiDimage.png)

创建完成后，点开```UGCObject```表，在此处创建物品信息，首先创建三个物品作为抽奖奖品，添加三条数据：1001，1002，1003。

![image.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/UaSq3image.png)

如图所示，每个物品可配置以下信息：
- 项目ItemID：物品的索引ID
- 物品名称：物品对外展示的名称
- 物品概述：物品对外展示的介绍
- 小Icon：物品对外展示的Icon

---

### 3. 创建抽奖货币

在创建完成三个奖品后，还需要创建对应的用于抽奖的抽奖货币，其本质是一个物品，通过商品的形式扣费。

首先需在物品表中创建一个ID为 `1004` 的物品作为抽奖货币。

![image.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/I1c4Eimage.png)

创建完成后，将抽奖货币添加至商品表作为商品，打开`UGCShop`表，添加一个商品，并在 `物品ID` 处填入抽奖货币的物品ID，即`1004`，此处的商品ID即为抽奖货币的正式ID。

![image.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/UjUFwimage.png)

---

### 4. 创建奖池表和掉落表

模板分别需要使用两张配置表：奖池表和掉落表。其中奖池表主要配置奖池信息，如奖池名，抽奖价格等，掉落表则与奖池表一一对应，主要配置该奖池可以抽取的奖品。

**配置掉落表**

> 掉落表的具体配置方式可以参考 [通用掉落](https://developer.gp.qq.com/wikieditor/#/catalog/20058)

打开掉落表（UGCDrop），插入一条数据，作为一个奖池的奖品配置，其中掉落ID为此条数据的索引，在配置奖池表时，我们需要将此ID填入，记录此处ID为0。

![image.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/y0J4Timage.png)

在添加奖励物品信息之前，首先，需要创建三个独立的物品，创建 ```物品表``` (Product Table)，插入三条数据，如图所示：

![image.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/oFnwcimage.png)

回到 ```掉落表```， 选择物品信息，我们插入三条物品信息，物品ID填写物品表中创建的物品ID，玩家开启抽奖，将会从这三个物品中随机获得。

![image.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/oBl6Kimage.png)

**配置奖池表**

``UGCLottery`` 抽奖表包含了奖池名称、买赠礼包、抽奖规则等相关配置，是抽奖模板的核心配置表之一，各配置项的使用方式将在后文【抽奖配置】中详细说明。
其中，```DrawCostID``` 为消耗货币的ID，本质为物品，通过商品的形式进行扣费，需按照上述方式使用物品表与商品表进行创建，创建完成后将抽奖货币对应的商品ID配置于此处。

![image.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/BUanCimage.png)

如图所示，在奖池表中配置了以下数据：

- ID：奖池索引，唯一
- OverrideDropID：掉落物品池ID，此处为0，即为上方掉落表中配置的掉落物品信息
- DrawCostID：抽奖所用的货币ID，此处为9000000，即上方商品表中配置的与抽奖货币相对应的商品ID
- OneDrawCostNum：单次抽奖所需花费的货币
- TenCostNum：十次抽奖所需花费的货币
- Name：奖池名称
- Icon：奖池Icon
- GiftProgressRewards：累次抽奖奖励，此处我们配置了10次抽奖奖励，可以获取五个1002物品。

> 创建方式：右键 ``各种各样的 -> 选择数据表格``，结构体选择 ``Lottery Data`` ，若结构体选择列表中没有该选项，需先打开一次结构体资源 ``Lottery Data`` 。抽奖表的名称必须为 ``UGCLottery``

## 抽奖配置

### 标题配置

奖池标题包含奖池名称与奖池说明的配置：

- 奖池名称：UGCLottery抽奖表的 ``Name`` 字段
- 奖池说明：UGCLottery抽奖表的 ``LotteryRule`` 字段

---

### 抽奖货币配置

一个奖池只允许使用一种货币抽取，各类货币都是物品，通过商品的形式进行扣费，因此在配置抽奖货币之前，需先基于 [物品表](https://developer.gp.qq.com/wikieditor/#/catalog/20008?autoJump=%E7%89%A9%E5%93%81%E8%A1%A8%EF%BC%88UGCObject%EF%BC%89) 创建物品，创建完成后将抽奖货币添加至 [商品表](https://developer.gp.qq.com/wikieditor/#/catalog/20008?autoJump=%E5%95%86%E5%93%81%E8%A1%A8%EF%BC%88UGCShop%EF%BC%89) ，将商品ID填入UGCLottery抽奖表的 ``DrawCostID`` 字段。

---

### 奖池页签配置

每个奖池都可以配置名称和icon图，通过页签的形式切换不同的奖池，名称与icon复用标题配置的字段：

- 页签名称：UGCLottery抽奖表的 ``Name`` 字段
- 页签icon：UGCLottery抽奖表的 ``Icon`` 字段

**ShowAward**

### 抽奖规则配置

抽奖模板支持了包括首抽免费、保底、限抽、折扣等丰富的抽奖规则，均以参数化的配置方式在UGCLottery表中实现。

**价格设定**

抽奖形式分为单抽与十连抽，每种抽奖消耗的货币价格分开设定。

- 单抽：``OneDrawCostNum`` 为单抽价格的设定字段<br>
![image.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/GT7Dhimage.png)
- 十连抽：``TenDrawCostNum `` 为十连抽价格的设定字段<br>
![image.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/Xzrbvimage.png)

**奖池关联**

由于抽奖模板使用掉落的形式决定抽奖标的，因此奖池即为掉落ID指向的物品库，掉落表的说明可查看 [通用掉落](https://developer.gp.qq.com/wikieditor/#/catalog/20058) 部分内容：
- OverrideDropID：掉落表ID，与DropGroupID互斥
- DropGroupID：掉落组表ID，与OverrideDropID互斥

![image.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/g7YuOimage.png)

**每日限抽**

如果 ``DailyDrawLimit`` 填写了>0的值，则代表开启每日限抽功能，限抽规则为每日0点重置抽奖次数，0代表无限制；
``DailyDrawGroup`` 可以设定奖池组，同一组内的奖池将共享累抽次数，0代表不属于任何奖池。

![image.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/90MgAimage.png)

**保底奖励**

保底奖励分为首抽保底和十连抽保底，奖励也使用掉落的形式产出，开发者可以根据需要进行配置。

【首抽保底】

![image.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/oETDDimage.png)

- OverrideFirstDrawGuarantDropID：掉落表ID，与FirstDrawGuarantDropGroupID互斥
- FirstDrawGuarantDropGroupID：掉落组表ID，与OverrideFirstDrawGuarantDropID互斥

``FirstDrawGuarantResetType`` 支持对首抽的保底奖励按规则进行重置：
- NotReset：不重置，一次性的保底奖励
- DailyReset：每日重置，每天零点进行重置
- MonthlyReset：每月重置，每月一号零点进行重置
- WeeklyReset：每周重置，每周周一零点进行重置

【十连抽保底】

![image.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/QgwW3image.png)

- OverrideGuarantDropID：掉落表ID，与GuarantDropGroupID互斥
- GuarantDropGroupID：掉落组表ID，与OverrideGuarantDropID互斥

> 十连抽保底为默认每次都有保底奖品，因此不具备保底重置概念

**首抽折扣**

如果开发者希望对首次抽奖打折甚至免费，可以勾选 ``IsFirstDrawDiscountOpen`` 以启用首抽折扣功能

**注意：**
> 首抽折扣目前仅对单抽生效

![image.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/DIOasimage.png)

【折扣值】

配置 ``FirstDrawDiscountCost`` 具体的折扣数值，此处为打完折之后的价格，例如原价100，折后80，则应填写80。

【首抽折扣重置】

``FirstDrawDistcountResetType`` 支持对首抽折扣进行重置：
- NotReset：不重置，一次性的首抽折扣
- DailyReset：每日重置，每天零点进行重置
- MonthlyReset：每月重置，每月一号零点进行重置
- WeeklyReset：每周重置，每周周一零点进行重置

## 注意事项

- 目前不支持直接消耗绿洲币进行抽奖，抽奖只消耗自定义货币
- 抽奖时货币不足会弹出商城购买商品的窗口，需要按照商城模板的步骤正确配置
- 每次抽奖不支持掉落0个或多个物品
- 买赠礼包如果单进度配置了多项物品时，只会在界面中显示第一个物品的icon图及数量信息

# 礼包模板

开发者可以将商品、货币或者其他各类道具整合成礼包的形式进行促销售卖，也可以作为游戏内刺激玩家持续追逐目标的激励物，礼包模板提供了便捷的配置方案，支持常规礼包与自选礼包两种礼包形式，是适用于 [商城](https://developer.gp.qq.com/wikieditor/#/catalog/20026)、[抽奖](https://developer.gp.qq.com/wikieditor/#/catalog/20027)、[签到](https://developer.gp.qq.com/wikieditor/#/catalog/20060) 等多种应用场景的通用解决方案。

<br>

## 模板结构

礼包分为常规礼包和自选礼包两类，不同类型礼包的模板结构略有差异。

### 常规礼包

玩家无法在常规礼包中主动选择奖励物品，当使用礼包后即获得礼包中的全部物品。

![QQ20241115-21289-HD.gif](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/uikY6QQ20241115-21289-HD.gif)

**使用礼包界面**

当打开礼包时会自动弹出使用礼包的二次确认界面，该界面上会展示礼包相关的详情信息。

![image.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/Qdognimage.png)

**A. 礼包弹窗标题**

配置礼包使用时的弹窗标题（非礼包名称），字数限制14个字以内

**B. 礼包详情**

礼包自身也是一种物品，与普通物品一样具有物品名称、物品描述及物品的icon等属性

【礼包名称】

配置在 [物品表](https://developer.gp.qq.com/wikieditor/?timeStamp=1725589224096#/catalog/20008?autoJump=%E7%89%A9%E5%93%81%E8%A1%A8%EF%BC%88UGCObject%EF%BC%89) 中的物品名称，字数限制7个字以内

【礼包描述信息】

配置在 [物品表](https://developer.gp.qq.com/wikieditor/?timeStamp=1725589224096#/catalog/20008?autoJump=%E7%89%A9%E5%93%81%E8%A1%A8%EF%BC%88UGCObject%EF%BC%89) 中的物品描述，字数限制同物品描述

【礼包库存】

当前已拥有的礼包数量

【礼包icon图】

配置在 [物品表](https://developer.gp.qq.com/wikieditor/?timeStamp=1725589224096#/catalog/20008?autoJump=%E7%89%A9%E5%93%81%E8%A1%A8%EF%BC%88UGCObject%EF%BC%89) 中的小icon，尺寸规格：55*55 

**C. 数量选择器**

模板自带数量选择器，当库存数量>1时可以选择当次需要使用的礼包数量，此时获得的物品数量为 **``使用的礼包数量 × 单礼包中各物品的数量``**

---

### 自选礼包

自选礼包支持玩家单次选择其中一份奖励，因此比常规礼包多一级交互，当玩家选择了目标奖励物品后才获得对应的物品。

**可选物品界面**

弹窗中展示礼包下所有的可选物品，各物品包含 [物品表](https://developer.gp.qq.com/wikieditor/?timeStamp=1725589224096#/catalog/20008?autoJump=%E7%89%A9%E5%93%81%E8%A1%A8%EF%BC%88UGCObject%EF%BC%89) 中配置的物品名称、物品icon图及设定的物品数量，其中物品名称默认只展示7个字长度的内容，超出部分以 ...替代，长按物品可显示完整名称及物品描述信息。

## 礼包配置

# 签到模板

签到功能通过奖励机制来增强玩家参与游戏的动力与黏性，是提升玩家活跃度和留存率的有效手段之一，签到模板为开发者提供快速添加签到功能的途径。

<br>

## 签到概述

![image.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/G65l8image.png)

**签到周期**

签到属于一种活动类型，因此需要设定活动的开始时间与结束时间，但是签到周期不一定等同于活动周期，可以将有效的签到时间范围视为活动时间段的子集，未到达或超出活动时间都无法正常签到。

**签到规则**

为了避免玩家刻意追逐高价值奖励、影响活跃与留存的效果，签到的形式统一为累计签到，每天只能签到并领取奖励一次；如果签到中断，下次参与活动按已累计天数顺序继续签到。

**签到类型**

模板支持重置型与非重置型两类签到：

- 非重置型：一次性的签到活动，为了确保通用性，仅提供7天和14天两种模板
- 重置型：支持周签到与月签到
	- 周签到：自然周每周一5:00重置活动与签到进度
	- 月签到：自然月每月1日5:00重置活动与签到进度

**补签规则**

模板支持补签，开发者可以设定允许补签的天数及补签消耗的道具，规则上先签到后补签，即玩家必须完成当天的签到后才能执行补签动作。

<br>

# 虚拟物品

[商业化系统](https://developer.gp.qq.com/wikieditor/#/catalog/20008) 引入了物品与商品的概念，商品是物品被附加了商业化属性后的特殊形式，因此物品是商业化功能中的基础要素，当玩家在游戏中产生物品数据的交互时，通过虚拟物品管理物品数据的流转变化。

<br>

## 虚拟物品概述

不同于 [物资系统](https://developer.gp.qq.com/wikieditor/#/catalog/20101) 中具备场景具象化、可与背包/仓库交互特征的实体物品，基于 [物品表](https://developer.gp.qq.com/wikieditor/#/catalog/20008?autoJump=%E7%89%A9%E5%93%81%E8%A1%A8%EF%BC%88UGCObject%EF%BC%89) 定义创建的物品仅有概念层面的业务含义，没有与场景和背包的数据交互，因此属于一类虚拟物品。

使用物品表的功能都涉及到虚拟物品的应用，例如 [商城模板](https://developer.gp.qq.com/wikieditor/#/catalog/20026) 中购买商品后即以虚拟物品的形式获得该物品，又如 [抽奖模板](https://developer.gp.qq.com/wikieditor/#/catalog/20027) 中抽取的奖品也是以虚拟物品的形式发放到玩家账户等等，这些功能通过 ``物品ID + 数量`` 的表结构形式表示虚拟物品的实例数据，同时虚拟物品与背包物品之间也允许存在单向的映射关系。

![image.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/xDG30image.png)

<br>

## 虚拟物品管理

由于虚拟物品应用场景的通用性，对虚拟物品的功能进行了 [GamePart模块化](https://developer.gp.qq.com/wikieditor/#/catalog/20090) 封装，开发者可以调用虚拟物品GamePart提供的API实现对虚拟物品数据的管理。

### 启用虚拟物品GamePart

在编辑器菜单栏上点击【玩法通用设置】按钮，在 ``DA_GameModeGeneral`` 数据资产蓝图的 ``Active Game Part Configs`` 属性下添加“GP_VirtualItemManager”模块，以启用虚拟物品的模块功能。

![image.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/2FpkJimage.png)

---

### 管理虚拟物品数据

虚拟物品GamePart通过 [``VirtualItemManager``](https://developer.gp.qq.com/api/#/searchContent/VirtualItemManager?classDetailShow=true&path=class%2Fdetail%2F%E5%92%8C%E5%B9%B3%E5%85%A8%E5%B1%80%E6%8E%A5%E5%8F%A3%2F%E5%95%86%E4%B8%9A%E5%8C%96%E4%B8%8E%E5%8A%9F%E8%83%BD%E6%A8%A1%E6%9D%BF%2FVirtualItemManager.json&isSelect=1&apiEnc=%5B%22%E7%B1%BB%22%5D&apiLabel=VirtualItemManager) 类函数提供数据管理的API，与常规的实体物品一样支持添加、查询、扣除等基础操作，有所区别的是虚拟物品的数据默认支持持久化存储，因此玩家拥有的虚拟物品可以跨对局存在。

以添加虚拟物品场景为例，调用 [``AddVirtualItem``](https://developer.gp.qq.com/api/#/searchContent/VirtualItemManager?classDetailShow=true&path=class%2Fdetail%2F%E5%92%8C%E5%B9%B3%E5%85%A8%E5%B1%80%E6%8E%A5%E5%8F%A3%2F%E5%95%86%E4%B8%9A%E5%8C%96%E4%B8%8E%E5%8A%9F%E8%83%BD%E6%A8%A1%E6%9D%BF%2FVirtualItemManager.json&isSelect=1&apiEnc=%5B%22%E7%B1%BB%22%5D&apiLabel=VirtualItemManager&autoJump=AddVirtualItem) 可以给玩家发放指定的虚拟物品，通过 [``GetOwnedVirtualItems``](https://developer.gp.qq.com/api/#/searchContent/VirtualItemManager?classDetailShow=true&path=class%2Fdetail%2F%E5%92%8C%E5%B9%B3%E5%85%A8%E5%B1%80%E6%8E%A5%E5%8F%A3%2F%E5%95%86%E4%B8%9A%E5%8C%96%E4%B8%8E%E5%8A%9F%E8%83%BD%E6%A8%A1%E6%9D%BF%2FVirtualItemManager.json&isSelect=1&apiEnc=%5B%22%E7%B1%BB%22%5D&apiLabel=VirtualItemManager&autoJump=GetOwnedVirtualItems) 可以查询玩家当前所拥有的所有虚拟物品。

**代码示例**

```lua
-- DS发放虚拟物品
local GamePartActor = UGCGamePartSystem.GetGamePartGlobalActor("VirtualItemManager");
if GamePartActor then
	GamePartActor:AddVirtualItem(self, 1001, 3);
end

-- 客户端查询拥有的虚拟物品数据
local GamePartActor = UGCGamePartSystem.GetGamePartGlobalActor("VirtualItemManager");
if GamePartActor then
	local OwnedItems = GamePartActor:GetOwnedVirtualItems()
  log_tree(OwnedItems)
end 
```

## 虚拟物品应用

### 功能模板中的虚拟物品

虚拟物品被广泛应用在使用了物品表的各类功能模板中，例如商城中购买或者签到获取的礼包，即是以虚拟物品的形式发放到玩家账户中，而礼包打开后获得的子物品也同样是虚拟物品，因此通过功能模板获得的物品默认都持久存储在玩家的账户中，需要通过 [``VirtualItemManager``](https://developer.gp.qq.com/api/#/searchContent/VirtualItemManager?classDetailShow=true&path=class%2Fdetail%2F%E5%92%8C%E5%B9%B3%E5%85%A8%E5%B1%80%E6%8E%A5%E5%8F%A3%2F%E5%95%86%E4%B8%9A%E5%8C%96%E4%B8%8E%E5%8A%9F%E8%83%BD%E6%A8%A1%E6%9D%BF%2FVirtualItemManager.json&isSelect=1&apiEnc=%5B%22%E7%B1%BB%22%5D&apiLabel=VirtualItemManager) 提供的API管理玩家的虚拟物品数据。

> 商城模板使用的GamePart模块默认引用了虚拟物品依赖，因此可以理解为从商城购买获得的物品实质均为虚拟物品

![image.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/UQ2p7image.png)

### 虚拟物品转换背包物品

> <span style="color:red;font-weight:bold;">当前版本该功能存在bug，在背包满格的时候无法正常转换数据，建议开发者临时通过 [``RemoveVirtualItem``](https://developer.gp.qq.com/api/#/searchContent/VirtualItemManager?classDetailShow=true&path=class%2Fdetail%2F%E5%92%8C%E5%B9%B3%E5%85%A8%E5%B1%80%E6%8E%A5%E5%8F%A3%2F%E5%95%86%E4%B8%9A%E5%8C%96%E4%B8%8E%E5%8A%9F%E8%83%BD%E6%A8%A1%E6%9D%BF%2FVirtualItemManager.json&isSelect=1&apiEnc=%5B%22%E7%B1%BB%22%5D&apiLabel=VirtualItemManager&autoJump=RemoveVirtualItem) 和 [``AddItemV2``](https://developer.gp.qq.com/api/#/searchContent/UGCBackpackSystemV2?classDetailShow=true&path=class%2Fdetail%2F%E5%92%8C%E5%B9%B3%E5%85%A8%E5%B1%80%E6%8E%A5%E5%8F%A3%2F%E7%89%A9%E5%93%81%E4%B8%8E%E8%83%8C%E5%8C%85%2FUGCBackpackSystemV2.json&isSelect=1&apiEnc=%5B%22%E7%B1%BB%22%5D&apiLabel=UGCBackpackSystemV2&autoJump=AddItemV2) 组合实现等同效果，等后续版本修复后再启用</span>

当功能模板中的物品涉及物资系统的物品时，存在虚拟物品向背包物品数据转换的需求场景。例如每日签到领取枪械，为实现该功能开发者需要监听获得虚拟物品的委托，调用 [``AddItemV2``](https://developer.gp.qq.com/api/#/searchContent/UGCBackpackSystemV2?classDetailShow=true&path=class%2Fdetail%2F%E5%92%8C%E5%B9%B3%E5%85%A8%E5%B1%80%E6%8E%A5%E5%8F%A3%2F%E7%89%A9%E5%93%81%E4%B8%8E%E8%83%8C%E5%8C%85%2FUGCBackpackSystemV2.json&isSelect=1&apiEnc=%5B%22%E7%B1%BB%22%5D&apiLabel=UGCBackpackSystemV2&autoJump=AddItemV2) 发放枪械物品，同时触发 [``OnAddItemV2``](https://developer.gp.qq.com/api/#/searchContent/UBackpackComponentV2?classDetailShow=true&path=class%2Fdetail%2FOthers%2FUBackpackComponentV2.json&isSelect=1&apiEnc=%5B%22%E7%B1%BB%22%5D&apiLabel=UBackpackComponentV2&autoJump=OnAddItemV2) 回调后，再通过 [``RemoveVirtualItem``](https://developer.gp.qq.com/api/#/searchContent/VirtualItemManager?classDetailShow=true&path=class%2Fdetail%2F%E5%92%8C%E5%B9%B3%E5%85%A8%E5%B1%80%E6%8E%A5%E5%8F%A3%2F%E5%95%86%E4%B8%9A%E5%8C%96%E4%B8%8E%E5%8A%9F%E8%83%BD%E6%A8%A1%E6%9D%BF%2FVirtualItemManager.json&isSelect=1&apiEnc=%5B%22%E7%B1%BB%22%5D&apiLabel=VirtualItemManager&autoJump=RemoveVirtualItem) 移除对应的虚拟物品。为了简化这类场景的实现，提供了配表的转换方式。

**创建物品映射表**

当启用虚拟物品GamePart时，会自动在 ``Asset/Data/Table`` 目录下生成名为“UGCObjectMapping”的物品映射表：

如果缺失该映射表，可以通过编辑器菜单栏 ``窗口 -> Create System Default Table -> Create VirtualItem Table`` 重新创建物品表与映射表；或者在 ``Asset/Data/Table`` 目录下右键 ``各种各样 -> 数据表格`` 基于 ``UGCVirtual Item Mapping Data`` 结构体创建该映射表。

![image.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/fQWpuimage.png)
![image.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/g2qqSimage.png)

**配置物品映射关系**

打开 ``UGCObjectMapping`` 映射表，分别填入 [物品表](https://developer.gp.qq.com/wikieditor/#/catalog/20008?autoJump=%E7%89%A9%E5%93%81%E8%A1%A8%EF%BC%88UGCObject%EF%BC%89) 中定义的物品ID与对应的物资系统中的 [物品ID](https://developer.gp.qq.com/wikieditor/#/catalog/20101?autoJump=%E5%9F%BA%E7%A1%80%E9%85%8D%E7%BD%AE)，系统将为任意途径获取的虚拟物品自动完成向背包数据的转换，相关物品将进入背包，且持久化存储该物品，对虚拟物品的数量更新、移除等操作也会自动同步至背包物品。

> 虚拟物品与背包物品的映射转换为单向1:1的绑定关系，即只存在虚拟物品转为背包物品，且一种虚拟物品对应一种背包物品

## 玩法特权接入流程

### 设计玩法特权

开发者设计玩法特权时，可包含但不限于以下几类权益类型：
- 道具权益：玩家可获得特定的道具奖励，例如特权玩家每日登陆领取福利。
- 商城权益：商城(限战斗内)购买相关的特权，例如额外的限购条件、折扣价格等。
- 资源加速权益：战斗中获取资源的效率更高，例如击杀敌人的经验增益、宝箱开启加速等。
- 其他权益：击杀特效、枪口特效等外显型效果。

---

特权的配置主要通过 [绿洲启元管理平台](https://gpeditor.qq.com/auth?authsrc=user#myprojects) 完成。登录账号后，在【项目管理】的【玩法项目列表】中，点击对应项目的【创建玩法内特权】即可配置特权。

> 特权需要单独开启权限才能正常使用，可联系绿洲官方开启权限。

- 勾选绑定的特权内容将在玩法版本上架后于游戏内展示，未勾选则不展示。

### 实现特权功能

#### 获取玩家特权信息

识别玩家特权身份，可调用 [GetPlayerPrivilegesInformation](https://developer.gp.qq.com/api/#/searchContent/UGCPrivilegeSystem?classDetailShow=true&path=class%2Fdetail%2F%E5%92%8C%E5%B9%B3%E5%85%A8%E5%B1%80%E6%8E%A5%E5%8F%A3%2F%E5%95%86%E4%B8%9A%E5%8C%96%E4%B8%8E%E5%8A%9F%E8%83%BD%E6%A8%A1%E6%9D%BF%2FUGCPrivilegeSystem.json&isSelect=1&apiEnc=%5B%22%E7%B1%BB%22%5D&apiLabel=UGCPrivilegeSystem&autoJump=GetPlayerPrivilegesInformation) 获取玩家所拥有的管理端配置的特权ID信息列表，通过信息列表判定玩家是否拥有特定特权。特权id的范围区间为1001-9999，特权的详细内容可在绿洲启元管理平台查看。

---

#### 识别玩家特权身份

在各权益的触发场景下，通过调用 [DoesPlayerHavePrivileges](https://developer.gp.qq.com/api/#/searchContent/UGCPrivilegeSystem?classDetailShow=true&path=class%2Fdetail%2F%E5%92%8C%E5%B9%B3%E5%85%A8%E5%B1%80%E6%8E%A5%E5%8F%A3%2F%E5%95%86%E4%B8%9A%E5%8C%96%E4%B8%8E%E5%8A%9F%E8%83%BD%E6%A8%A1%E6%9D%BF%2FUGCPrivilegeSystem.json&isSelect=1&apiEnc=%5B%22%E7%B1%BB%22%5D&apiLabel=UGCPrivilegeSystem&autoJump=DoesPlayerHavePrivileges) 判断玩家的特权身份。

---

#### 执行特权逻辑

特权逻辑需由开发者根据权益类型自行实现。例如定义一个玩家登录领取金币的特权，则需要在触发玩家登录事件时，判断玩家特权身份，对拥有特权的玩家执行发放金币的逻辑。
特殊地，对于商城类型的权益，开发者可以在商品表中单独配置一类特权商品，同时提供了 [BuyUGCPrivilegeCommodity2](https://developer.gp.qq.com/api/#/searchContent/UGCCommoditySystem?classDetailShow=true&path=class%2Fdetail%2F%E5%92%8C%E5%B9%B3%E5%85%A8%E5%B1%80%E6%8E%A5%E5%8F%A3%2F%E5%95%86%E4%B8%9A%E5%8C%96%E4%B8%8E%E5%8A%9F%E8%83%BD%E6%A8%A1%E6%9D%BF%2FUGCCommoditySystem.json&isSelect=1&apiEnc=%5B%22%E7%B1%BB%22%5D&apiLabel=UGCCommoditySystem&autoJump=BuyUGCPrivilegeCommodity2) 购买接口，对特权玩家的购买行为调用此接口实现。购买的回调可绑定 [BuyUGCCommodityResultDelegate](https://developer.gp.qq.com/api/#/searchContent/UGCCommoditySystem?classDetailShow=true&path=class%2Fdetail%2F%E5%92%8C%E5%B9%B3%E5%85%A8%E5%B1%80%E6%8E%A5%E5%8F%A3%2F%E5%95%86%E4%B8%9A%E5%8C%96%E4%B8%8E%E5%8A%9F%E8%83%BD%E6%A8%A1%E6%9D%BF%2FUGCCommoditySystem.json&isSelect=1&apiEnc=%5B%22%E7%B1%BB%22%5D&apiLabel=UGCCommoditySystem) 。

> 部分曾使用 [BuyUGCCommodity](https://developer.gp.qq.com/api/#/searchContent/UGCCommoditySystem?classDetailShow=true&path=class%2Fdetail%2F%E5%92%8C%E5%B9%B3%E5%85%A8%E5%B1%80%E6%8E%A5%E5%8F%A3%2F%E5%95%86%E4%B8%9A%E5%8C%96%E4%B8%8E%E5%8A%9F%E8%83%BD%E6%A8%A1%E6%9D%BF%2FUGCCommoditySystem.json&isSelect=1&apiEnc=%5B%22%E7%B1%BB%22%5D&apiLabel=UGCCommoditySystem&autoJump=BuyUGCCommodity) 购买接口的老玩法，可以调用 [BuyUGCPrivilegeCommodity](https://developer.gp.qq.com/api/#/searchContent/UGCCommoditySystem?classDetailShow=true&path=class%2Fdetail%2F%E5%92%8C%E5%B9%B3%E5%85%A8%E5%B1%80%E6%8E%A5%E5%8F%A3%2F%E5%95%86%E4%B8%9A%E5%8C%96%E4%B8%8E%E5%8A%9F%E8%83%BD%E6%A8%A1%E6%9D%BF%2FUGCCommoditySystem.json&isSelect=1&apiEnc=%5B%22%E7%B1%BB%22%5D&apiLabel=UGCCommoditySystem&autoJump=BuyUGCPrivilegeCommodity) 兼容接口。

## 注意事项

1. 所有玩法特权不可直接修改战斗属性。
2. 玩法特权提供的道具、资源需控制在合理范围内，不允许特权玩家与免费玩家存在过大差距。
3. 开发者需要把特权方案发送平台审核（即进行【特权绑定】），审核通过后放可接入玩法特权。
4. 开发者需在玩法详情页自主更新、维护特权说明，特权内容调整后，必须同步平台并修改玩法详情页。

# 结算模板

结算模板是为游戏开发者提供的一套完整的对局结算解决方案，包含结算界面展示、数据统计、胜负判定、结算动画等核心功能。该模板支持单挑、小队对抗、小队排名板和个人排名板四种主流结算模式，帮助开发者快速实现游戏结算系统。

<br>

### 结算数据配置

结算界面的数据需通过调用BeginSettlement接口进行传入。主要包含以下两个数据表：PlayerResults（玩家结算数据）和 TeamResults（队伍结算数据）。

#### PlayerResults（玩家结算数据）

PlayerResults的类型为table，每一项对应一名玩家的全部结算数据。每项数据包含PlayerKey和Results，PlayerKey的类型为number，对应该玩家的Key；Results的类型为table，保存该玩家的各项结算数据，其中数据的类型均为string。

![企业微信截图_17592180107823.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/Oouei%E4%BC%81%E4%B8%9A%E5%BE%AE%E4%BF%A1%E6%88%AA%E5%9B%BE_17592180107823.png)

Results中的键名必须与蓝图“BP_GameResult”中【Scoreboard Headers】、【Player Result Headers】和【Player Info Result Headers】属性组内配置的Key字段相匹配，以确保数据能被正确读取并显示。

![结算模板绘图-数据2.drawio.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/wKllq%E7%BB%93%E7%AE%97%E6%A8%A1%E6%9D%BF%E7%BB%98%E5%9B%BE-%E6%95%B0%E6%8D%AE2.drawio.png)
![结算模板绘图-数据.drawio.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/pwo9I%E7%BB%93%E7%AE%97%E6%A8%A1%E6%9D%BF%E7%BB%98%E5%9B%BE-%E6%95%B0%E6%8D%AE.drawio.png)

若【Scoreboard Type】选择了TeamLeaderBoard，PlayerResults表应按玩家排名从高到低进行排列，即排名第一的玩家结算数据置于首位，排名最低的玩家结算数据置于末位。

---

#### TeamResults（队伍结算数据）

TeamResults的类型为table，每一项对应一个队伍的结算数据。每项数据包含TeamID和DeterminingResult，TeamID的类型为number，对应该队伍的ID；DetermingResult的类型为string，为队伍胜负判定数据，该字段的值将显示在结算界面的“胜负判定依据”区域。

![image.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/sN9lkimage.png)
![结算模板绘图-位置演示.drawio.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/XksFx%E7%BB%93%E7%AE%97%E6%A8%A1%E6%9D%BF%E7%BB%98%E5%9B%BE-%E4%BD%8D%E7%BD%AE%E6%BC%94%E7%A4%BA.drawio.png)

根据【Scoreboard Type】的不同，TeamResults的传入数据规则如下：
- 若【Scoreboard Type】为Duel或SquadDuel，且调用BeginSettlement时bDraw参数设为false，则将获胜方数据置于第一项，失败方数据置于第二项
- 若【Scoreboard Type】为TeamLeaderBoard，TeamResults表应按队伍排名从高到低进行排列，即排名第一的队伍结算数据置于首位，排名最低的队伍结算数据置于末位
- 若【Scoreboard Type】为SoloLeaderboard，队伍结算数据须为空。

``` lua
--Main.lua部分代码

--UI控件的初始化函数
function MainUI:Construct()
	--给按键绑定点击事件回调
	self.ButtonBeginSettlement.OnClicked:Add(self.ButtonBeginSettlementOnClicked, self);
	self.ButtonEndSettlement.OnClicked:Add(self.ButtonEndSettlementOnClicked, self);
end

function MainUI:ButtonBeginSettlementOnClicked()
	--获取PlayerController
	local PlayerController = UGCGameSystem.GetLocalPlayerController();
	--调用PlayerController的ServerRPC函数
	UnrealNetwork.CallUnrealRPC(PlayerController, PlayerController, "BeginSettlement");
	return nil;
end

function MainUI:ButtonEndSettlementOnClicked()
	--获取PlayerController
	local PlayerController = UGCGameSystem.GetLocalPlayerController();
	--调用PlayerController的ServerRPC函数
	UnrealNetwork.CallUnrealRPC(PlayerController, PlayerController, "EndSettlement");
	return nil;
end

--UGCPlayerController.lua部分代码

--接收RPC调用
function UGCPlayerController:GetAvailableServerRPCs()
	return
  "BeginSettlement",
  "EndSettlement"
end

function UGCPlayerController:BeginSettlement()
	--获取所有的PlayerController
  local AllPlayerController = UGCGameSystem.GetAllPlayerController();

  --构造玩家结算数据
  local PlayerResults = {
		--第一位玩家的数据
		{
			PlayerKey = UGCGameSystem.GetPlayerKeyByPlayerController(AllPlayerController[1]);
			Results = {
				Kill = "35",
        Death = "5",
        Assist = "0",
        Time = "1分30秒",
        Val5 = "123",
        Val6 = "321",
        Val7 = "132",
        Val8 = "923",
        Val9 = "292"
			}
		},
    --第二位玩家的数据
		{
			PlayerKey = UGCGameSystem.GetPlayerKeyByPlayerController(AllPlayerController[2]);
			Results = {
				Kill = "25",
        Death = "20",
        Assist = "0",
        Time = "1分20秒",
        Val5 = "123",
        Val6 = "321",
        Val7 = "132",
        Val8 = "923",
        Val9 = "292"
			}
		}
	};

	--构造队伍结算数据
  local TeamResults = {
		--第一位玩家所在的队伍为胜利方
    {
    	TeamID = UGCPlayerControllerSystem.GetTeamID(AllPlayerController[1]);
      DetermingResult = "40";
    },
    --第二位玩家所在的队伍为失败方
    {
    	TeamID = UGCPlayerControllerSystem.GetTeamID(AllPlayerController[2]);
      DetermingResult = "30";
    }
	};

	--开始结算
  GameResultManager:BeginSettlement(PlayerResults,TeamResults,false);
end

function UGCPlayerController:EndSettlement()
	--结束结算
  GameResultManager:EndSettlement();
end
```

# 任务模板

任务模板是为游戏开发者提供的一套完整的任务系统解决方案，包含界面展示、任务逻辑、奖励发放等核心功能，模板预置了多种常见任务类型（如击杀、收集、在线时长），并支持成长型与活跃型两种主流任务线模式，能帮助开发者快速构建规范的任务体系，显著提升开发效率。

<br>

## 功能概要

任务模板功能包含任务读取、任务进度更新、奖励领取等部分，这些功能的执行流程分别如下：
- 任务读取时，系统读取绑定的任务线表，并依据其中存储的任务ID生成对应的任务实例，最终在任务界面中向玩家展示
- 任务进度更新中，当玩家进行游戏时，系统会实时更新玩家数据，并自动同步更新所有关联任务的进度状态，无需玩家手动操作
- 奖励领取功能包括任务奖励领取和活跃度奖励领取共两种，所有领取的奖励均以物品形式发放至玩家账户

## 功能调试

可为控件蓝图的对应按钮添加以下逻辑，实现任务界面入口的功能。

> 控件蓝图路径：ExtendResource/TaskTemplate/OfficialPackage/Asset/Task/Blueprint/WBP_TaskMainUIButton

![ScreenShot_2025-10-16_162048_904.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/5alahScreenShot_2025-10-16_162048_904.png)

将控件通过 [AddToViewport](https://developer.gp.qq.com/api/#/searchContent/UUserWidget?classDetailShow=true&path=class%2Fdetail%2FOthers%2FUUserWidget.json&isSelect=1&apiEnc=%5B%22%E7%B1%BB%22%5D&apiLabel=UUserWidget&autoJump=AddToViewport) 添加到屏幕视口中，启动PIE调试后出现测试UI界面。

```lua

--UGCGameState.lua部分代码
function UGCGameState:ReceiveBeginPlay()
```

## 1. 概述

UGC 商业化系统中的**补偿机制**用于在特定场景下向玩家补发商品（物品）。系统提供两个层次的补偿广播委托：

| 委托 | 补偿方式 | 生效范围 | 适用场景 |
|------|----------|----------|----------|
| `CompensateUGCCommodityDelegate` | 单个商品补偿 | 服务器 & 客户端 | 单笔补偿，如异常恢复、单商品补发 |
| `CompensateUGCCommodityBatchDelegate` | 批量商品补偿 | 仅服务器 | 多笔补偿，如活动奖励、批量异常恢复 |

> **注意**：补偿操作本身由系统**自动完成**，这两个委托仅用于**广播补偿结果**，供开发者监听并执行后续逻辑（如日志记录、UI提示、数据同步等）。

## 4. API 详细说明

### 4.1 `UGCCommoditySystem.CompensateUGCCommodityDelegate`

**委托类型**：`Delegate`

**广播时机**：
- 系统对玩家进行**单笔商品补偿**后
- 补偿操作已完成且数据已更新

**参数说明**：

| 参数 | 类型 | 说明 |
|------|------|------|
| `PlayerKey` | `number` | 玩家 PlayerKey，服务器端标识 |
| `UID` | `number` | 玩家 UID |
| `CommodityID` | `number` | 物品 ID，对应 `UGCObject` 配置表 |
| `Count` | `number` | 补偿数量（正整数） |
| `ProductID` | `number` | 商品 ID，对应 `UGCShop` 配置表 |

**生效范围**：服务器 & 客户端（两端都会收到）

**适用场景**：
- 单个商品异常补发
- 单笔退款补偿
- 单个活动奖励补发

---

### 4.2 `UGCCommoditySystem.CompensateUGCCommodityBatchDelegate`

**委托类型**：`Delegate`

**广播时机**：
- 系统对玩家进行**批量商品补偿**后
- 所有补偿项处理完成后一次性触发

**参数说明**：

| 参数 | 类型 | 说明 |
|------|------|------|
| `PlayerKey` | `number` | 玩家 PlayerKey，服务器端标识 |
| `UID` | `number` | 玩家 UID |
| `CommodityList` | `table` | 补偿商品列表（数组结构） |

**`CommodityList` 结构说明**：

```lua
CommodityList = {
    {
        CommodityID = 1001,  -- 物品ID（对应UGCObject表）
        Count       = 5,     -- 物品数量（正整数）
        ProductID   = 900001 -- 商品ID（对应UGCShop表）
    },
    {
        CommodityID = 1002,
        Count       = 3,
        ProductID   = 900002
    },
    -- ... 更多商品项
}
```

**生效范围**：仅服务器

**适用场景**：
- 活动批量奖励发放
- 运营一次性多道具补偿
- 版本更新统一补偿

---

### 4.3 核心差异对比

| 特性 | `CompensateUGCCommodityDelegate` | `CompensateUGCCommodityBatchDelegate` |
|------|----------------------------------|---------------------------------------|
| **补偿方式** | 单个商品补偿 | 批量商品补偿 |
| **参数结构** | 单个商品详细信息（5个独立参数） | 商品列表（`CommodityList` 数组） |
| **生效范围** | 服务器 & 客户端 | 仅服务器 |
| **触发频率** | 每补偿一个商品触发一次 | 批量补偿完成后触发一次 |
| **典型用途** | 单笔异常恢复 | 活动/运营批量补发 |

## 5. 使用示例

### 5.1 监听单笔补偿

```lua
-- 在 Server 或 Client 端注册监听（建议在玩家初始化时注册）
function MySystem:InitializeCompensateListener()
    -- 监听单笔补偿
    UGCCommoditySystem.CompensateUGCCommodityDelegate:Add(self, self.OnCompensateUGCCommodity)
end

---单笔补偿回调
---@param PlayerKey number
---@param UID number
---@param CommodityID number
---@param Count number
---@param ProductID number
function MySystem:OnCompensateUGCCommodity(PlayerKey, UID, CommodityID, Count, ProductID)
    print_dev(StringFormat_Dev("[MySystem] 收到单笔补偿 UID=%d CommodityID=%d Count=%d ProductID=%d", 
        UID, CommodityID, Count, ProductID))
    
    -- 示例：记录补偿日志
    self:LogCompensation(PlayerKey, UID, CommodityID, Count, ProductID)
    
    -- 示例：客户端弹提示（仅客户端需要）
    if not UGCGameSystem.IsServer() then
        self:ShowCompensationTip(CommodityID, Count)
    end
end
```

### 5.2 监听批量补偿（仅 Server）

```lua
-- 仅在服务器端注册
function MySystem:InitializeBatchCompensateListener()
    if not UGCGameSystem.IsServer() then
        return
    end
    
    -- 监听批量补偿
    UGCCommoditySystem.CompensateUGCCommodityBatchDelegate:Add(self.OnCompensateUGCCommodityBatch, self)
end

---批量补偿回调
---@param PlayerKey number
---@param UID number
---@param CommodityList table
function MySystem:OnCompensateUGCCommodityBatch(PlayerKey, UID, CommodityList)
    print_dev(StringFormat_Dev("[MySystem] 收到批量补偿 UID=%d 商品数量=%d", 
        UID, #CommodityList))
    
    -- 遍历补偿列表处理每一项
    for _, item in ipairs(CommodityList) do
        print_dev(StringFormat_Dev("[MySystem] 补偿项 CommodityID=%d Count=%d ProductID=%d", 
            item.CommodityID, item.Count, item.ProductID))
        
        -- 示例：写入数据库或运营日志
        self:RecordCompensationToDB(UID, item.CommodityID, item.Count, item.ProductID)
    end
    
    -- 示例：发送运营事件上报
    self:ReportCompensationEvent(UID, CommodityList)
end
```

### 5.3 注销监听

## 6. 常见问题

### Q1：补偿操作由谁发起？开发者需要调用补偿接口吗？

**A**：补偿指令由**商业化后台**下发到系统，自动完成数据更新并广播委托。开发者**不需要**手动调用补偿相关接口，只需**监听委托**获取补偿结果通知即可。

### Q2：两个委托的参数中都有 `CommodityID` 和 `ProductID`，它们有什么区别？

**A**：

| 字段 | 含义 | 对应配置表 |
|------|------|-----------|
| `CommodityID` | 物品的 ID | `UGCObject` 表（物品定义） |
| `ProductID` | 商品的 ID | `UGCShop` 表（商品/售卖定义） |

一个 `ProductID`（商品）可能对应一个或多个 `CommodityID`（物品），补偿时通常两者都会提供。

### Q3：客户端能收到批量补偿委托吗？

**A**：不能。`CompensateUGCCommodityBatchDelegate` **仅在服务器端广播**。如果客户端需要感知批量补偿，可以通过以下方式：
- 监听 `UGCCommoditySystem.UGCCommodityPlayerDataChangedDelegate`（物品变化时广播）
- 或服务器收到批量补偿后，由服务器逻辑主动下发自定义通知到客户端。

### Q4：补偿的数据会保存到结算存档吗？

**A**：会。补偿数据会持久化到玩家的结算存档中，确保跨局数据一致性。

### Q5：补偿委托与物品变化委托有什么关系？

**A**：补偿委托（`CompensateUGCCommodityDelegate` / `CompensateUGCCommodityBatchDelegate`）在补偿**完成后立即触发**，专注于补偿这一特定场景；而 `UGCCommodityPlayerDataChangedDelegate` 在**玩家物品数据发生任何变化时**触发，范围更广。两者可同时监听，用途不同。

### Q6：PIE 模式下如何测试补偿功能？

**A**：PIE 模式下系统不会连接真实商业化后台，开发者可以通过直接调用注册的函数，模拟测试补偿的发放。

## 核心概念

### 商品系统

- **商品 (Product)**: 可购买的商品实体，关联到商品表 [`UGCShop`](https://developer.gp.qq.com/wikieditor/#/catalog/20008?autoJump=%E5%95%86%E5%93%81%E8%A1%A8%EF%BC%88UGCShop%EF%BC%89)，包含商品ID、名称、价格、折扣等信息
- **物品 (Commodity)**: 玩家已购买的物品，存储在 `UGCCommodityComponent` 的 `UGCCommodityPlayerData` 中
- **货币 (Ticket)**: 绿洲币，用于购买商品的通用货币
- **启元币 (ActiveCoin)**: 特殊货币，用于特定场景

```lua
        print(string.format("商品ID: %s, 已购买: %d", product.ProductID, product.BuyProductLimitCount))
    end
end
```

## 关键要点

### 购买相关

- ✅ 必须使用折后价格：购买时传入的 `Cost` 参数必须通过 `GetSellingPriceAfterDiscount()` 获取，不能直接使用 `UGCShop` 表的配置价格
- ✅ 协议注册：使用协议方式购买前，必须先在服务器调用 `RegBuyCMD()` 注册
- ✅ 余额检查：系统会自动检查玩家绿洲币余额，余额不足时显示充值提示
- ✅ 二次确认：购买时会显示二次确认弹窗，展示商品信息和总价
- ⚠️ 充值配置：绿洲币充值功能需要配置 `UGCOasisTokenInGamePurchaseAndRecharge` 参数

---

### 使用相关

- ✅ 协议注册：使用协议方式使用前，必须先在服务器调用 `RegUseCMD()` 注册
- ✅ 数量校验：使用数量必须是正整数，否则会返回错误
- ✅ 二次确认可选：客户端使用可设置 `bShowDialog` 控制是否显示确认弹窗
- ⚠️ 服务器调用：服务器调用 `UseUGCCommodity2()` 时必须传入 `PlayerController`，否则无效

---

### 数据查询

- ✅ `PlayerKey`处理：服务器调用查询接口必须传入 `PlayerKey`，客户端可省略
- ✅ 空值处理：如果玩家控制器无效，返回 0 或 nil
- ✅ 跨局同步：跨局物品变化通过 `BuyUGCCommodityResultBetweenGamesDelegate` 广播

---

### 价格计算

- ✅ 折扣时间判断：系统会自动判断当前服务器时间是否在折扣有效期内
- ✅ 取整处理：折后价格使用 `math.floor()` 取整
- ✅ 原价返回：不在折扣时间内返回 `UGCShop` 表的 `SellingPrice`

## 开发者注意事项

### 购买流程注意事项


⚠️ **充值提示触发条件**：

- 玩家在非灰度的预发布或者正式服，并且玩法开启了充值权限时，会显示充值提示
- 如果充值功能关闭，直接返回余额不足错误

⚠️ **灰度用户处理**：

- 灰度用户不显示充值提示，直接返回余额不足错误
- 可通过 `PlayerController.IsGray` 判断是否灰度用户

---

### 使用流程注意事项

⚠️ **服务器调用限制**：

- 服务器调用 `UseUGCCommodity2()` 时会忽略 `bShowDialog` 参数
- 服务器调用必须传入有效的 `PlayerController`

⚠️ **物品数量检查**：

- 系统不会在客户端检查玩家是否拥有足够的物品数量
- 数量校验由服务器端完成

---

### Delegate 绑定时机

✅ **推荐绑定位置**：

- 购买/使用结果：在 UI 初始化时绑定
- 数据变化：在 `PlayerController:ReceiveBeginPlay()` 中绑定
- 跨局变化：在 `PlayerController:ReceiveBeginPlay()` 中绑定

⚠️ **避免重复绑定**：

- 同一个 `Delegate` 不要多次绑定
- 使用前可以先用 `:Unbind()` 解除之前的绑定

---

### 性能优化建议

✅ **减少查询频率**：

- 商品价格可以缓存，不要每次购买都查询
- 物品列表变化通过 `Delegate` 监听，不要轮询查询

✅ **批量操作**：

- 批量购买建议使用多个 `PromiseFuture` 并行处理

⚠️ **避免频繁调用**：

- `GetUGCCommodityList()` 在短时间内多次调用返回相同结果
- 通过 `UGCCommodityPlayerDataChangedDelegate` 监听变化

# 活动模板

UGC活动模板是和平精英绿洲起源编辑器提供的官方活动系统，允许创作者快速配置和管理游戏内的活动任务和签到功能，无需编写复杂的代码即可实现完整的活动逻辑。

活动模版当前支持配置两种活动类型：【任务型活动】和【签到型活动】。

活动标题、副标题、活动背景图、任务类型、任务奖励、任务重置类型、签到天数、是否支持补签、签到奖励等均在编辑器表格内配置，活动开始时间、活动结束时间在管理端配置。


<br>

## 功能概述

活动模板系统通过配置驱动的方式，为UGC工程提供任务型和签到型两种标准活动类型。系统由三个核心模块组成：配置层（数据表格）、逻辑层（ActivityManager）、表现层（UI组件）。创作者只需在编辑器中配置相关表格和UI资源，系统会自动处理任务进度跟踪、奖励发放、签到状态管理等功能。

- 对于重置型榜单，玩家通过 ``历史榜单数据`` 界面领取奖励，且根据重置类型历史榜单将额外保留展示一天/一周/一个月

H1. 签到天数

签到活动的天数，签到天数上限为7天

H2. 签到奖励

每日签到获得的奖励道具，奖励数量上限为1个道具

H3. 签到及补签规则（和签到模版规则一致）

H3.1-签到规则

领奖为累签领奖方式，与自然日期无对应关系，按排列顺序依次领奖，每天领取一次。若某日没有登录领奖，不会跳过，再登录时按上次领奖进度，继续领奖即可

H3.2-补签规则

1. 所有签到活动支持补签，允许补签天数以及消耗道具支持表格配置
2. 若触发补签，则在本日应该签到的奖励后，加上x天可补签的奖励并显示补签按钮，x取决与配置
3. 先签到后补签，补签位置永远在本日该签到奖励后，与天数无绑定关系，若玩家只签到不补签，则第二天可签到和补签奖励依次向后移动

H3.3-签到状态

【已签到】：已完成签到的天数显示该按钮
【签到】：可签到的天数显示该按钮
【待签到】：尚未签到且不能补签的天数，显示该按钮
【补签】：已签到后点击补签按钮，弹补签窗口，消耗对应道具进行补签；若未完成本日签到，点击补签按钮弹出提示文字 “本日签到后才能补签！”
【待补签】：完成上一个可补签日，才能补签下一个；若未完成可补签，点击【待补签】弹出提示文字“上一个补签日完成后才能补签！”

### GM调试

为便于进行功能调试，开发者可通过配置通用GM面板，添加一个打开通行证界面的快捷入口，具体配置方法请参阅 [扩展GM功能](https://developer.gp.qq.com/wikieditor/#/catalog/20109?autoJump=%E6%89%A9%E5%B1%95GM%E5%8A%9F%E8%83%BD) 。

``` Lua
local ProjGM = {}
local FreezeExpSwitch = false

function ProjGM:Register(DebugUI)
    local UGCGMUI = require("client.ingame.ugc.ugc_gmui")

    local CurFuncList = {}
```
